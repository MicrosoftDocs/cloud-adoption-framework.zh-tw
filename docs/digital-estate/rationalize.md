# <a name="rationalize-the-digital-estate"></a>合理化數位資產

雲端合理化是評估資產以判斷何種方法最適合用來將其裝載於雲端的程序。 在您決定 [方法](./approach.md) 並匯總 [清查](./inventory.md)之後，就可以開始進行雲端合理化。 雲端合理化討論最常見的合理化選項。

## <a name="traditional-view-of-rationalization"></a>合理化的傳統檢視

當您將傳統的合理化程式視覺化為複雜的決策樹時，很容易瞭解合理化。 數位資產中的每個資產都會透過一個程式來送出，以產生五個 (的) 的合理化。 此程序非常適用於小型資產。 針對較大的資產，它的效率不佳，而且可能會導致嚴重的延遲。 我們將檢查該程序以了解原因。 然後，我們會產生更有效率的模型。

**清查：** 使用傳統模型完成完整的合理化，需要完整清查資產，包括應用程式、軟體、硬體、作業系統和系統效能度量。

**量化分析：** 在決策樹中，量化問題推動了第一層的決策。 常見的問題包括下列各項：

- 資產目前是否使用中？
- 若是如此，它是否已最佳化並適當調整大小？
- 資產之間有哪些相依性？ 這些問題對於清查的分類而言是很重要的。

**定性分析：** 下一組決策需要以定性分析形式的人類智慧。 通常，此處所提出的問題對解決方案是唯一的，而且只能由商務專案關係人和 power users 回答。 這些決策通常會延遲進程，因此會大幅降低。 這種分析通常會在每個應用程式中耗用40至 80 FTE 小時。

如需有關建立定性分析問題清單的指引，請參閱 [數位資產規劃的方法](./approach.md)。

**合理化決策：** 在經驗豐富的合理化小組手中，定性和量化資料可建立明確的決策。 只可惜，合理化經驗豐富的小組僱用成本高昂，或是需要好幾個月的事前訓練。

## <a name="rationalization-at-enterprise-scale"></a>企業規模的合理化

如果這項投入時間相當耗時，而且對 50-VM 數位資產來說相當困難，請想像在具有上千部 Vm 和數百個應用程式的環境中推動業務轉型所需的工作。 所需的人力工作很容易就能超過 1500 FTE 小時和九個月的規劃。

雖然完整合理化是結束狀態和移入的絕佳方向，但很少會產生高 ROI (報酬率) 相對於所需的時間和能源。

當合理化對於財務決策很重要時，值得考慮專門參與雲端合理化的專業服務組織來加速流程。 就算如此，完整的合理化也可能是成本高昂且耗時的工作，因而延遲轉換或業務成果。

本文的其餘部分將說明另一種方法，也稱為增量合理化。

## <a name="incremental-rationalization"></a>漸進式合理化

大型數位資產的完整合理化很容易發生風險，可能會因為其複雜性而受到延遲。 在累加式方法背後的假設是，延遲的決策會錯開企業的負載，以降低障礙的風險。 經過一段時間後，此方法會建立一個有機模型，以開發更有效率地達成合格合理化決策所需的程式和體驗。

### <a name="inventory-reduce-discovery-data-points"></a>清查：減少探索資料點

少陣列織會投資時間、能源和費用，以維持完整數位資產的精確即時清查。 遺失、遭竊、重新整理週期和員工上線通常會證明使用者裝置的詳細資產追蹤。 在傳統的內部部署資料中心內維護精確的伺服器和應用程式清查的 ROI 通常很低。 大部分的 IT 組織都有更緊急的解決問題，而不是追蹤資料中心內固定資產的使用量。

在雲端轉換中，清查直接與營運成本相關聯。 必須要有精確的清查資料，才能完成妥善規劃。 可惜的是，目前的環境掃描選項可能會延遲數周或數個月的決策。 幸運的是，有幾個技巧可以加速資料收集。

代理程式型掃描是最常被提到的延遲。 傳統合理化所需的強大資料通常只會隨著每個資產上執行的代理程式一起收集。 這項代理程式的相依性通常會減緩進度，因為它可能需要來自安全性、作業和管理功能的意見反應。

在增量合理化程式中，無代理程式解決方案可用於初始探索，以加速早期決策。 根據環境中的複雜度層級，可能仍需要以代理程式為基礎的解決方案，但它可以從關鍵的商務變更路徑中移除。

### <a name="quantitative-analysis-streamline-decisions"></a>量化分析：簡化決策

不論清查探索的方法為何，量化分析都可以推動初始決策和假設。 尤其是在嘗試找出第一個工作負載時，或合理化的目標是高階成本比較時，更是如此。 在增量合理化流程中，雲端策略小組和雲端採用小組會將五個 [合理化的合理化](./5-rs-of-rationalization.md) 限制為兩個精確的決策，並只套用這些量化因素。 這可簡化分析，並減少推動變更所需的初始資料量。

例如，如果組織在 IaaS 遷移至雲端的過程中，您可以假設大部分的工作負載都會淘汰或重新裝載。

### <a name="qualitative-analysis-temporary-assumptions"></a>定性分析：暫時性假設

藉由減少潛在的結果數目，就可以更輕鬆地達到資產未來狀態的初始決定。 當您減少選項時，您也可以在這個初期階段減少業務要求的問題數目。

例如，如果選項受限於重新裝載或淘汰，企業只需要在初始合理化期間回答一個問題，也就是是否要淘汰資產。

「分析建議沒有任何使用者主動使用此資產。 這是正確的，還是忽略了什麼？」 這類二進位問題通常會比透過定性分析更容易執行。

這個精簡方法會產生基準、財務計劃、策略和方向。 在後續的活動中，每個資產都會經歷進一步的合理化和定性分析，以評估其他選項。 您在此初始合理化中所做的所有假設都會先經過測試，再遷移個別的工作負載。

## <a name="challenge-assumptions"></a>挑戰假設

前一節的結果是已滿假設的粗略合理化。 接下來，我們可以挑戰其中的某些假設。

### <a name="retire-assets"></a>淘汰資產

在傳統的內部部署環境中，裝載小型、未使用的資產鮮少會對年度成本造成重大影響。 有幾個例外狀況，分析和淘汰實際資產所需的 FTE 工作，就超過剪除和淘汰這些資產的成本節約。

當您移至雲端帳戶處理模型時，淘汰資產可能會大幅節省年度營運成本和預先進行的遷移工作。

組織在完成量化分析之後淘汰20% 或更多的數位資產，並不罕見。 建議您在採取動作之前，先進行更定性的分析。 確認後，淘汰這些資產可以產生雲端遷移的第一個 ROI。 這通常是最大的節省成本因素之一。 因此，雲端策略小組應該透過執行 [遷移方法](../migrate/index.md)的方式，同時監督資產的驗證和淘汰，以達成早期的財務勝利。

### <a name="program-adjustments"></a>方案調整

很少有公司只進行一個轉型程序。 在降低成本、追求市場成長與新增收益來源之間所做的抉擇，鮮少是二元決策。 因此，我們建議雲端策略小組與 IT 合作，找出在主要轉型旅程範圍之外之平行轉換工作的資產。

在本文所提供的 IaaS 遷移範例中：

- 要求 DevOps 團隊找出已是部署自動化一部分的資產，並從核心遷移計畫中移除這些資產。

- 要求資料和 R&D 個小組找出正在為新收入串流進行電源的資產，並將其從核心遷移計畫中移除。

這項以程式為焦點的定性分析可快速執行，並跨多個遷移待處理專案建立對齊。

您可能還需要將某些資產視為重新裝載資產一段時間。 您稍後可以在初始遷移之後進行合理化。

## <a name="select-the-first-workload"></a>選取第一個工作負載

實作第一個工作負載是測試和學習的關鍵。 這是示範和建立成長思維的第一次機會。

### <a name="business-criteria"></a>商務準則

為確保業務透明度，請識別雲端策略小組的業務單位成員所支援的工作負載。 最好選擇一個小組有有意的工作，以及將動機移至雲端的動機。

### <a name="technical-criteria"></a>技術準則

選取相依性最低、且能夠以小型資產群組的形式移動的工作負載。 建議您選取具有已定義測試路徑的工作負載，讓驗證變得更容易。

第一個工作負載通常會部署在實驗環境中，且不具操作或管理能力。 請務必選取不會與安全資料互動的工作負載。

### <a name="qualitative-analysis"></a>定性分析

雲端採用小組和雲端策略小組可以共同合作來分析此小型工作負載。 這種共同作業建立了一個建立和測試定性分析準則的控制機會。 較小型的人口會讓您有機會調查受影響的使用者，並在一周或更短的時間內完成詳細的定性分析。 如需常見的定性分析因素，請參閱 [五個合理化 Rs](./5-rs-of-rationalization.md)中的特定合理化目標。

### <a name="migration"></a>移轉

雲端採用小組可以與持續合理化平行地開始遷移小型工作負載，以在下列關鍵領域中擴展學習：

- 利用雲端提供者的平臺強化技能。
- 定義符合長期願景所需的核心服務和 Azure 標準。
- 更進一步瞭解作業在轉換之後可能需要變更的方式。
- 瞭解任何固有的商業風險，以及企業對於這些風險的容忍度。
- 根據業務的風險承受度，為治理建立基準或最小可行產品 (MVP) 。

## <a name="release-planning"></a>版本規劃

當雲端採用小組正在執行第一個工作負載的遷移或執行時，雲端策略小組可以開始排列其餘應用程式和工作負載的優先順序。

### <a name="power-of-10"></a>10的乘冪

傳統的方法可合理化嘗試符合所有可預見的需求。 所幸，要開始進行一個轉型程序，通常不需要為每個應用程式設定一個計劃。 在累加式模型中，10種方法的強大功能可提供良好的起點。 在此模型中，雲端策略小組會選取要遷移的前10個應用程式。 這十個工作負載應會同時包含簡單和複雜的工作負載。

### <a name="build-the-first-backlogs"></a>建立第一個待處理專案

雲端採用小組和雲端策略小組可以共同處理前10個工作負載的定性分析。 這種努力會建立第一個優先順序的遷移待辦專案，以及第一個優先的發行待辦專案。 這種方法可讓小組逐一查看方法，並提供足夠的時間來建立適當的程式以進行定性分析。

### <a name="mature-the-process"></a>成熟的流程

在兩個小組同意定性分析準則之後，評定可能會成為每個反復專案中的工作。 達到評估準則的共識通常需要兩到三個版本。

在評量移至遷移的累加執行程式之後，雲端採用小組可以更快反覆運算評定和架構。 在這個階段中，雲端策略小組也會抽象化，減少時間的清空。 這也可讓雲端策略小組專注于優先處理尚未存在於特定版本的應用程式，以確保與不斷變化的市場狀況緊密一致。

並非所有已排定優先順序的應用程式都已可進行移轉。 當小組進行更深入的定性分析，並探索可能會提示 reprioritization 待處理專案的商務事件和相依性時，排序可能會變更。 某些版本可能會將少量的工作負載群組在一起。 有些則可能只包含單一工作負載。

雲端採用小組可能會執行反復專案，而不會產生完整的工作負載遷移。 工作負載越小，相依性就越少，工作負載可納入單一短期衝刺或反覆執行中的可能性就越高。 基於這個理由，我們建議發行待處理專案中的前幾個應用程式很小，並包含一些外部相依性。

## <a name="end-state"></a>最終狀態

經過一段時間之後，雲端採用小組和雲端策略小組會一起完成清查的完整合理化。 這種漸進式方法可讓小組在合理化流程中持續更快。 它也可協助轉換旅程更快地產生有形的商務結果，而不需要事先進行分析。

在某些情況下，財務模型可能太過嚴格，而不會進行其他合理化。 在這種情況下，您可能需要更傳統的合理化方法。
