# <a name="understand-and-align-the-portfolio-hierarchy"></a>瞭解並對齊組合階層

商務需求通常是透過資訊技術支援、改善或加速。 提供已定義之商業價值的技術集合稱為_工作負載_。 該集合可能包括應用程式、伺服器或虛擬機器、資料、裝置，以及其他類似群組的資產。

一般來說，商務專案關係人和技術領導人會共同負責每個工作負載的持續支援。 在工作負載生命週期的某些階段中，系統會清楚指出這些角色。 在工作負載生命週期的更多操作階段中，這些角色可能會轉換為共用的 operations management 小組或雲端作業小組。 當工作負載數目增加時，角色（指定或隱含）會變得更複雜且更 matrixed。

大部分的企業都依賴多個工作負載來提供重要的商務功能。 工作負載、資產和支援因素（專案、人員、流程和投資）的集合稱為「_組合_」。 商務、開發和營運人員的矩陣需要一個組合階層，以顯示工作負載和支援服務如何彼此配合。

本文提供組合階層層級的清楚定義。 本文會將每一層中具有適當責任的各種小組結合在一起，以及該小組的最佳指引來源，以提供該層級的預期。 在本文中，階層的每個層級也稱為「_範圍_」。

## <a name="portfolio-hierarchy"></a>組合階層

工作負載及其支援的資產是任何組合的核心。 下列其他範圍或層級定義了這些工作負載的查看方式，以及可能支援小組的矩陣所影響的範圍。

當您要部署第一個工作負載時，工作負載和其資產可能是唯一定義的範圍。 其他層可能會在部署更多工作負載時明確定義。

![組合、平臺、登陸區域、工作負載和資產的階層。 在嵌套的父子式關聯性中，從最高到最低的順序列出。](../../_images/ready/hierarchy.png)

雖然條款可能有所不同，但所有的 IT 解決方案都包含資產和工作負載：

- **資產：** 支援工作負載或解決方案的技術功能最小單位。
- **工作負載：** 企業 IT 支援的最小單位。 工作負載是資產（基礎結構、應用程式和資料）的集合，可支援常見的商業目標或執行一般商務程式。

當公司透過 matrixed 方法或集中式方法來支援工作負載時，可能會有更廣泛的階層可支援這些工作負載：

- **登陸區域：** 登陸區域提供工作負載，可存取支援一或多個工作負載所_需的所有__基礎公用程式_（或共用配量）。
- **基本公用程式：** 這些共用的 IT 服務是工作負載在技術和業務組合內運作的必要專案。
- **Platform foundation：** 此組織結構會集中基本解決方案，並協助確保所有登陸區域都會強制執行這些控制項。
- **雲端平臺：** 視支援完整_組合_的整體策略而定，客戶可能需要多個雲端平臺，其中包含不同的平臺基礎部署，以管理多個區域、混合式解決方案，或甚至是多重雲端解決方案。
- **組合：** 透過技術鏡頭，此組合是橫跨所有雲端平臺之工作負載、資產和支援資源的集合。 透過商務鏡頭，組合是專案、人員、程式和投資的集合，可支援及管理技術組合來推動業務成果。 這兩個鏡頭會一起捕捉到_組合_。

## <a name="hierarchy-accountability-and-guidance"></a>階層責任與指引

負責管理組合階層的每個層級的小組。 下圖顯示負責團隊與指導方針之間的對應，以支援其商務決策、技術決策和技術實現。

> [!NOTE]
> 下列清單中所述的小組可能是虛擬小組（v 小組）或個人。 對於此階層的某些變體，部分有責任的小組可以折迭，如後續責任變化中所述。

![與階層一致的責任](../../_images/ready/hierarchy-with-roles.png)

<!-- docsTest:ignore "Strategy and Plan methodologies" "Migrate and Plan methodologies" "Migrate and Innovate methodologies" -->

- **組合：** 雲端策略小組和卓越的雲端中心（CCoE）使用策略和計畫方法，來引導影響整體組合的決策。 雲端策略小組負責雲端組合階層的企業層級。 雲端策略小組也應通知有關環境、登陸區域和高優先順序工作負載的決策。
- **雲端平臺：** 雲端治理小組負責確保每個環境之間的一致性，以符合管控方法。 雲端治理小組負責管理所有環境中的所有資源。 雲端治理小組應針對可能需要例外狀況或變更來管理原則的變更進行諮詢。 雲端治理小組也應通知工作負載和資產採用的進度。
- **登陸區域和雲端基礎：** 雲端平臺小組負責開發支援採用的登陸區域和平臺公用程式。 雲端自動化小組負責自動化的開發，並持續支援這些登陸區域和平臺公用程式。 這兩個小組都使用現成的方法來引導執行。 這兩個小組都應該知道工作負載採用的進度，以及企業或環境的任何變更。
- **工作負載：** 採用會在工作負載層級發生。 雲端採用小組使用遷移與創新方法來建立可擴充的程式，以加速採用。 採用完成後，工作負載的擁有權可能會轉移至使用管理方法來引導操作管理的雲端作業小組。 這兩個小組都應該熟悉如何使用 Microsoft Azure 架構完善的架構，以進行詳細的架構決策，以影響其支援的工作負載。 這兩個小組都應該知道登陸區域和環境的變更。 這兩個小組有時可能會參與登陸區域功能。
- **資產：** 資產通常是雲端營運小組的責任。 該小組會使用管理方法中的管理基準來引導操作管理決策。 它也應該使用 Azure Advisor 和 Microsoft Azure 架構完善的架構，以根據作業需求來進行詳細的資源和架構變更。

### <a name="accountability-variants"></a>責任變體：

- **單一環境：** 當企業只需要一個環境時，通常不需要 CCoE。
- **單一登陸區域：** 如果環境只有單一登陸區域，則治理和平臺功能可能會合並成一個小組。
- **單一工作負載：** 有些企業在單一登陸區域和單一環境中只需要一個工作負載，或幾個工作負載。 在這些情況下，治理、平臺和營運團隊之間的責任區隔並不需要。

## <a name="common-workload-and-accountability-examples"></a>一般工作負載和責任範例

下列範例說明組合階層。

### <a name="cots-workloads"></a>COTS 工作負載

傳統上，企業採用商業現成的（COTS）軟體解決方案來處理商務程式。 這些解決方案會先安裝、設定完成，然後再進行操作。 在設定之後，解決方案架構的變更並不大。

在這些案例中，COTS 解決方案的任何雲端採用，會在轉換至雲端營運小組時結束。 然後，雲端操作小組會成為該軟體的技術擁有者，並承擔管理設定、成本、修補週期和其他營運需求的責任。

這些工作負載包括會計封裝、物流軟體或業界特定的解決方案。 在 Microsoft 的術語中，這些套件的廠商稱為獨立軟體廠商（Isv）。 許多 Isv 都提供一項服務，可在您的訂用帳戶中部署和維護其軟體套件的實例。 它們也可能提供在自己的雲端裝載環境中執行的軟體套件版本，並提供工作負載的平臺即服務（PaaS）替代方案。

除了 PaaS 供應專案以外，雲端作業小組會負責確保這些工作負載的基本營運合規性需求。 雲端營運小組應該與雲端治理小組合作，以配合成本、效能和其他架構要素。

### <a name="in-development-with-active-revisions"></a>使用作用中修訂進行開發

當 COTS 解決方案或 PaaS 供應專案未符合商務需求，或沒有解決方案存在時，企業會建立自訂開發的工作負載。 一般來說，IT 組合的小型百分比會使用此工作負載方法。 但是，這些工作負載通常會降低比例高的商業結果比重，特別是與新收益串流相關的結果。 這些工作負載很容易對應到新的創新想法。

針對以 agile 方法和 DevOps 實務為基礎的各種變動，這些工作負載優於傳統 IT 管理的商業/DevOps。 針對這些工作負載，雲端營運團隊可能不會有數年的交付。 在這些情況下，開發小組會做為工作負載的技術擁有者。

由於有大量時間和相關的資本限制，自訂開發選項通常僅限於高價值的機會。 一般範例包括應用程式創新、深度資料分析或任務關鍵性商務功能。

### <a name="breakfix-or-sunset-development"></a>中斷/修正或終止開發

在自訂開發的工作負載達到尖峰成熟度後，開發小組可能會重新指派給其他專案。 在這些情況下，技術擁有權通常會轉換成雲端營運小組。 當需要小型的修正時，營運小組會登記開發人員以解決錯誤。

在某些情況下，開發小組會移至最終會取代目前工作負載的專案。 或者，小組可能會繼續進行，因為工作負載所支援的商業機會會被淘汰。這些是日落案例的範例，其中雲端營運小組會作為技術擁有者，直到不再需要工作負載為止。

在這兩種情況下，雲端營運小組通常都是長期的技術擁有者和決策者。 當操作變更需要重大的架構變更時，該小組可能會登記應用程式開發人員。

### <a name="mission-critical-workloads"></a>任務關鍵性工作負載

在每個公司中，有幾個工作負載對企業而言太重要，而無法成功。 有了這些任務關鍵性工作負載，通常會有各種責任層級的營運和開發擁有者。 這些小組應配合操作變更和架構變更，將生產解決方案的中斷降至最低。

這些案例需要專注于職責的分離。 為了達成責任的區分，營運小組通常會在生產環境中保有日常作業變更的責任。 當這些操作變更需要架構變更時，非生產環境中的開發或採用小組會在作業小組將變更套用至生產環境之前完成。

具有必要責任劃分的任務關鍵性工作負載範例包括 SAP、Oracle 或其他企業資源規劃（ERP）解決方案的工作負載，其跨越公司中的多個業務單位。

## <a name="strategy-portfolio-alignment"></a>策略組合的對齊

請務必瞭解雲端採用工作的策略性目標，並對齊組合以支援該轉換。 一些常見的策略性組合對齊方式有助於塑造組合階層的結構。 下列各節提供組合階層的組合和影響的範例。

### <a name="innovation-or-development-led-portfolio"></a>創新或開發導向的組合

有些公司（特別是快速成長的新創企業）擁有較平均百分比的自訂開發專案。 在開發-繁重的組合中，環境、登陸區域和工作負載通常會進行壓縮，而 &mdash; 特定的工作負載可能會有特定的環境（生產或非生產環境）。 這會產生環境、登陸區域和工作負載之間的1:1 比例。

因為環境會裝載自訂解決方案，所以 DevOps 管線和應用層級的報告可能會取代作業和治理功能的需求。 對於這些客戶，可能會降低專注于營運、治理或其他支援角色。 同時也特別強調雲端採用和雲端自動化小組的責任。

**組合對齊：** IT 組合可能會專注于工作負載和工作負載擁有者，以驅動重要的架構決策。 在採用和操作活動期間，這些小組可能會在 Microsoft Azure 架構良好的架構指引中找到更多價值。

**界限定義：** 邏輯界限（即使在企業層級）可能會專注于生產和非生產環境分割。 公司軟體組合中的產品可能也會有清楚的分割。 有時候，開發與裝載的客戶實例可能也會有分割。

### <a name="operations-led-portfolio"></a>營運導向組合

具有更多既定 IT 營運團隊的跨國企業組織，通常在治理和營運上的焦點高於開發。 在這些組織中，較高百分比的工作負載通常會對應至 COTS 或中斷/修正類別，由雲端營運小組內的技術擁有者維護。

**組合對齊：** IT 組合將會對齊工作負載，但這些工作負載會對應到營運單位或業務單位。 可能也有關于資金的模型、產業或其他業務分割需求的組織。

**界限定義：** 登陸區域可能會將應用程式分組成應用程式 archetype，以在類似的分割中保持類似的作業。 環境可能會參考實體結構，例如資料中心、國家、雲端提供者區域或其他營運組織標準。

### <a name="migration-led-portfolio"></a>遷移導向的組合

類似于作業導向的組合，主要是透過遷移建立的公事包，會以導致現有資產遷移的特定商業驅動程式為基礎。 一般來說，資料中心是這些驅動程式中的最大因素。

**組合對齊：** IT 組合可能是工作負載對齊，但較可能的資產已對齊。 如果公司的歷程記錄中發生轉換至 IT 作業，許多使用中的資產可能無法輕易地對應至已獲得資助的工作負載。 在這些情況下，許多資產可能沒有定義的工作負載或清除的工作負載擁有者，直到遷移程式的延遲。

**界限定義：** 登陸區域可能會將應用程式群組為反映內部部署分割的界限。 雖然不是最佳做法，但環境通常會比對代表網路分割實務的內部部署資料中心名稱和登陸區域。 較好的作法是遵循更緊密配合操作導向組合的分割。

### <a name="governance-led-portfolio"></a>治理導向的組合

與治理小組的一致應該會儘早發生。 透過治理實務和雲端治理工具，組合和環境界限可以充分平衡創新、作業和遷移工作的需求。

**組合對齊：** 治理導向的組合傾向于包含資料點，以捕捉創新和作業的詳細資訊，例如工作負載、作業擁有者、資料分類和營運重要性。 這些資料點會建立組合的圓角觀點。

**界限定義：** 治理導向組合中的界限，傾向于透過創新來偏好作業，同時使用管理群組驅動的階層，其對應至業務單位和開發環境的準則。 在階層的每個層級中，雲端治理界限可能會有不同程度的原則強制執行，以允許開發和創意彈性。 同時，生產等級的需求也可以套用至生產訂用帳戶，以確保責任區隔和一致作業。
