# <a name="business-commitment-in-cloud-management"></a>雲端管理的業務承諾

定義商務承諾是平衡優先順序的練習。 其目標是要以可接受的營運成本來調整適當的作業管理層級。 找出餘額需要幾個資料點和計算，如下節所述。

![平衡成本與復原](../../_images/manage/business-commitment-scale.png)

對企業穩定性的承諾（透過技術復原或其他 SLA 影響）是商業理由決策。 對於環境中的大部分工作負載而言，雲端管理的基準層級就已足夠。 對於其他人來說，會因為任何業務中斷的潛在影響，而輕鬆增加 2 &ndash;4x 的成本。 本系列的前幾篇文章可協助您瞭解各種工作負載中斷的分類和影響。 本文將協助計算傳回的。 如下圖所述，在每個雲端管理層級中，都有變化的點數，其成本可能會比復原的增加更快。 這些變化點將會提示詳細的商務決策和業務承諾。

## <a name="determining-a-proper-commitment-with-the-business"></a>決定企業的適當承諾

針對組合中的每個工作負載，雲端營運小組和雲端策略小組應符合雲端營運小組直接提供的管理層級。

以下是在建立企業承諾時要保持一致的重要層面：

- IT 作業必要條件
- 管理責任
- 雲端租賃
- 軟成本因素
- 遺失避免 ROI
- 驗證管理層級

本文的其餘部分將概述這些準則，以協助進行決策。

## <a name="it-operations-prerequisites"></a>IT 作業必要條件

[Azure 管理指南](../azure-management-guide/index.md)概述 azure 中可用的管理工具。 在達到業務承諾之前，它應該決定可接受的標準層級管理基準，以套用至所有受控工作負載。 接著，它會根據 CPU 核心計數、磁碟空間和其他與資產相關的變數，計算 IT 組合中每個受控工作負載的標準管理成本。 它也會根據架構來估計每個工作負載的複合 SLA。

> [!TIP]
> IT 營運小組通常會針對初始複合 SLA 使用預設的最低99.9% 執行時間。 他們也可以選擇根據平均工作負載來標準化管理成本，特別是針對具有最低限度記錄和儲存需求的解決方案。 將幾個中等重要性工作負載的成本平均，可以提供初始交談的起點。

> [!TIP]
> 針對使用[Ops 管理活頁簿](https://raw.githubusercontent.com/microsoft/CloudAdoptionFramework/master/manage/opsmanagementworkbook.xlsx)規劃雲端管理的讀者，應更新 ops 管理欄位以反映這些必要條件。 這些欄位包括承諾用量層級、複合 SLA 和每月成本。 每月成本應該代表每月新增的作業管理工具成本。

在下列各節中，作業管理基準會做為初始起始點進行驗證。

## <a name="management-responsibility"></a>管理責任

在傳統的內部部署環境中，通常會假設管理環境的成本是 IT 營運所擁有的 sunk 成本。 在雲端中，管理是直接影響預算的有目的決策。 每個管理功能的成本可能會更直接針對部署至雲端的每個工作負載進行屬性化。 這種方式可提供更大的控制能力，但它確實為雲端營運小組和雲端策略小組建立了一項需求，必須先承諾遵守關於責任的合約。

組織也可以選擇將[其一些持續的管理功能外包給服務提供者](https://www.microsoft.com/cloud-adoption-framework-offers?ot=manage)。 這些服務提供者可以使用[Azure 燈塔](https://azure.com/lighthouse)，讓組織能夠更精確地控制授與資源的存取權，以及對服務提供者所執行的動作有更大的透明度。

**委派的責任：** 由於不需要集中化並採用作業管理的額外負荷，因此許多組織的 IT 營運會考慮新的方法。 其中一個常見的方法稱為委派責任。 在卓越模型的雲端中心，平臺作業和平臺自動化提供了自助管理工具，可供商務導向營運小組使用，而不受中央 IT 營運小組的影響。 這種方法可讓商務專案關係人完全掌控與管理相關的預算。 此外，它也可讓 CCoE 確保正確地執行一組最小的護欄。 在此模型中，它會作為 broker 和指南，協助企業做出明智的決策。 商務營運負責監督相依工作負載的日常作業。

**集中式責任：** 合規性需求、技術複雜性和某些共用服務模型可能需要中央 IT 模型。 在此模型中，它會繼續維持作業管理責任。 如此一來，環境設計、管理控制項和治理工具可能會集中管理和控制，這在做出管理承諾時，會限制商務專案關係人的角色。 但是，雲端方法的成本和架構可見度讓集中化 IT 能夠更輕鬆地為每個工作負載溝通成本和層級的管理。

**混合模型：** 分類是管理責任之混合模型的核心。 從內部部署轉換至雲端的公司，可能需要一段時間的內部部署優先作業模型。 其他具有嚴格合規性需求或依賴 IT 外包廠商之長期合約的公司，可能需要集中式作業模型。 儘管這類條件約束，企業還是需要創新。 當快速創新必須在中央 IT 集中化責任模型的過程中進行時，分類和混合模型可能會提供平衡。 在這種方法中，中央 IT 針對所有任務關鍵性或包含機密資訊的工作負載，提供集中式作業模型。 不過，工作負載的所有其他分類可能會放在專為委派責任而設計的雲端環境中。 集中式責任方法是做為一般作業模型。 然後，企業可以根據所需的支援和敏感度層級，靈活運用特定的作業模型。

第一個步驟是承諾採取責任措施，這會塑造下列承諾。

**哪一組織會負責此工作負載的日常作業管理？**

## <a name="cloud-tenancy"></a>雲端租賃

對於大部分的企業而言，當所有資產都位於單一租使用者時，管理就會變得更容易。 不過，某些組織可能需要維護多個租使用者。 [使用 Azure 燈塔集中管理作業](../centralize-operations.md)一文提供一些範例，其中企業可能需要多租使用者的 Azure 環境。

**此工作負載會與所有其他工作負載一起位於單一 Azure 租使用者嗎？**

## <a name="soft-cost-factors"></a>軟成本因素

本文的下一節將概述與管理程式和工具層級相關聯之傳回的比較方法。 在該區段結束時，每個分析的工作負載將會測量管理成本（相對於預測的業務中斷影響）。 這種方法可讓您更輕鬆地瞭解是否保證投資更豐富的管理方法。

執行數位之前，請務必查看軟成本因素。 軟成本因素會產生報酬率，但這項報酬很難透過直接的硬性成本節約來測量，這會在收益與損失語句中顯示。 軟成本因素很重要，因為它們可能表示需要投資較高的管理層級，而不是且謹慎。

這些因素的幾個範例包括：

- 每日使用工作負載（由面板或 CEO）。
- 依前_x%_ 的客戶使用工作負載，導致其他地方的收益增加。
- 對員工滿意度的影響。

建立承諾所需的下一個資料點是軟成本因素的清單。 這些不需要在這個階段記載，但商務專案關係人應該留意這些因素的重要性，並從下列計算中排除。

## <a name="calculate-loss-avoidance-roi"></a>計算遺失避免 ROI

在計算作業管理成本的相對傳回時，負責雲端作業的 IT 小組應完成上述必要條件，並針對所有工作負載採用最低層級的管理。

下一次承諾是由與基準管理的供應專案相關聯的成本業務所接受。

**企業是否同意投資基準供應專案，以符合雲端營運的最低標準？**

如果企業不同意該管理層級，則必須設計解決方案，讓企業能夠繼續進行，而不會影響其他工作負載的雲端作業。

如果企業想要比標準層多，則本節的其餘部分將有助於驗證該投資和相關聯的傳回（以避免損失的形式）。

### <a name="increased-levels-of-management-design-principles-and-service-catalog"></a>增加的管理層級：設計原則和服務類別目錄

針對受控解決方案，除了管理基準以外，還可以套用數個設計原則和範本解決方案。 可靠性和復原的每個設計原則都會增加工作負載的營運成本。 為了讓 IT 和企業同意這些額外的承諾，請務必瞭解可能會因增加的投資而避免的損失。

下列計算將逐步解說公式，以進一步瞭解損失與增加的管理投資之間的比較。 如需計算增加管理成本的指引，請參閱有關[工作負載自動化](./workload.md)和[平臺自動化](./platform.md)的文章。

> [!TIP]
> 針對使用[Ops 管理活頁簿](https://raw.githubusercontent.com/microsoft/CloudAdoptionFramework/master/manage/opsmanagementworkbook.xlsx)規劃雲端管理的讀者，應更新 Ops 管理欄位以反映每個交談。 這些欄位包括承諾用量層級、複合 SLA 和每月成本。 每月成本應該代表每月新增的作業管理工具成本。 更新之後，這些欄位會更新 ROI 公式和下列每個欄位。

**預估中斷 = （1-複合 SLA 百分比）/* 一年中的時*數 *

### <a name="standard-loss-impact"></a>標準遺失影響

標準遺失影響（活頁簿中的「標準影響」）會預測任何中斷的財務影響，假設「估計的中斷」預測證明正確。 若要計算此預測，而不使用活頁簿，請套用下列公式：

**標準影響 = 預估停機時間 @ 3 9 的執行時間/* 時間值影響**

當商務專案關係人選擇投資較高層級的管理時，這會做為成本的基準。

### <a name="composite-sla-impact"></a>複合 SLA 影響

複合 SLA 影響（在活頁簿中標題為「承諾用量層級影響」）會根據執行時間 SLA 的變更來提供更新的會計影響。 這可讓您比較這兩個選項的預估財務影響。 若要計算這項預測的影響，而不使用試算表，請套用下列公式。

**複合 SLA 影響 = 估計的中斷/* 時間/值影響**

值代表已變更的承諾用量層級和新的複合 SLA 應避免的可能損失。

### <a name="comparison-basis"></a>比較基礎

比較基礎會評估標準影響和複合 SLA 影響，以判斷哪一個最適合在傳回資料行中。
