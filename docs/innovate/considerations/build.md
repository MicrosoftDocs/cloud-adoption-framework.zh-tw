# <a name="build-with-customer-empathy"></a>以客戶理解的方式打造

「必要是發明的母親」。 這個句耳熟能詳諺語會捕捉人類精神的 indelibility，以及我們要創造的自然磁片磁碟機。 如 Oxford 英文字典中所述，「當某個東西的需求變得很重要時，您會被迫找出取得或達成的方式」。 少數會拒絕這些有關發明的通用真相。 不過，如 [數位經濟中的創新](./index.md)所述，創新需要發明和採用之間的平衡。

從這種比喻開始，創新來自于更擴充的系列。 *客戶的理解是創新的榮幸。* 建立驅動創新的解決方案時，必須有合法的客戶需要，才能讓客戶回頭解決重大的挑戰。 這些解決方案是根據客戶所需的資訊，而不是根據其想要或 whims。 為了找出其真正的需求，我們先從理解開始，深入瞭解客戶的體驗。 「理解」是許多工程師、產品經理、甚至企業領導人的 underdeveloped 技能。 幸運的是，雲端架構設計人員角色的多樣性互動和快速步調已開始培養這項技能。

為什麼要讓這項重要的理解？ 從最小可行產品的第一版 (MVP) 到市場等級解決方案的正式推出，客戶的理解可協助我們瞭解並分享客戶的體驗。 我們將協助我們建立更好的解決方案。 更重要的是，它可讓我們更妥善地將鼓勵採用的解決方案定位到。 在數位經濟中，可最容易體諒前客戶需求的人可以建立更好的未來，以重新定義和領導市場。

## <a name="how-to-build-with-empathy"></a>如何以理解方式打造

規劃本質上是定義假設的練習。 我們所計畫的計畫愈多，我們看到的假設也會蔓延到絕佳概念的基礎。 假設通常是自我理解的產品。 換句話說：「如果我在此位置，該怎麼辦？」 從組建階段開始，可將假設 invade 方案的期間降至最低。 這種方法也可加速與真實客戶的意見反應迴圈，並觸發先前的機會來學習及銳化理解。

> [!CAUTION]
> 正確定義組建的內容可能很棘手，需要一些作法。 如果您太快建立了某個東西，可能無法反映客戶的需求。 如果您花太多時間來瞭解初始客戶需求和解決方案需求，市場可能會在您有機會建立任何專案之前，先符合這些需求。 在任一案例中，學習的機會會大幅延遲或降低。 有時候，資料甚至可能已損毀。

歷程記錄中最創新的解決方案是以直覺的想法開始。 這項直覺感覺來自于現有的專業知識和第一手觀察。 我們從組建階段開始，因為它可讓您快速測試該直覺。 我們可以從該處養成更深入的瞭解，並更清楚的理解程度。 在每個反復專案或每次發行的解決方案中，都是建立 Mvp 來示範客戶的理解。

為了穩定此平衡動作，下列兩節將討論以理解和定義 MVP 建立的概念。

### <a name="define-a-customer-focused-hypothesis"></a>定義以客戶為焦點的假設

以明確的方式建立，根據定義的假設來建立解決方案，以說明特定客戶的需求。 下列步驟的目標是要制訂假設，以鼓勵採用的方式來建立。

1. 當您以理解的方式建立時，客戶一律是焦點。 這種意圖可能需要許多圖形。 您可以參考客戶原型、特定的角色，甚至是在您想要解決之問題的過程中，提供客戶的圖片。 請記住，客戶可以是內部 (員工或合作夥伴) 或外部 (消費者或企業客戶) 。 這項定義是第一個要測試的假設：我們可以協助這個特定的客戶嗎？
2. 瞭解客戶體驗。 以易懂的方式建立，您可以與客戶的經驗相關，並瞭解他們的挑戰。 這種思維方式表示要測試的下一個假設：我們是否能以這種可管理的挑戰協助此特定客戶？
3. 為單一挑戰定義簡單的解決方案。 依賴人員、流程和主題專家的專業知識，將會導致可能的解決方案。 這是要測試的完整假設：我們是否可以透過建議的解決方案，協助此特定客戶提供這項可管理的挑戰？
4. 抵達值語句。 您希望為這些客戶提供哪些長期的價值？ 這個問題的答案會建立您的完整假設：如何使用提議的解決方案解決這項可管理的挑戰，以改善這些客戶的生活？

最後一個步驟是理解導向的假設高潮。 它會定義物件、問題、解決方案，以及要進行改進的度量，而這一切都是在客戶的中心。 在量值和學習階段期間，應該測試每個假設。 客戶、問題陳述或解決方案中的變更，會因為團隊針對可定址的客戶群發展出更好的理解而預期。

> [!CAUTION]
> 其目標是要以客戶理解的方式 *建立* ，而不是使用它進行 *規劃* 。 在完美的客戶理解陳述的規劃和調整迴圈中，這一切都太簡單了。 在您嘗試開發這類語句之前，請先參閱下列各節，以瞭解如何定義和建立 MVP。

在證實核心假設之後，稍後的反復專案會將焦點放在更容易理解的測試以外的成長測試。 建立、測試及驗證之後，您就可以開始瞭解大規模市場的規模。 這可以透過擴展稍早所述的標準假設公式來完成。 根據可用的資料，預估總市場的大小 (潛在客戶) 的數目。

從該處，估計經歷類似挑戰且可能對此解決方案有興趣的總市場百分比。 這是您可定址的市場。 下一個要測試的假設是：使用建議的解決方案來解決這項可管理的挑戰，將如何改進 *x%* 的客戶生活？ 客戶的小型取樣會顯示領先的指標，以建議對參與的客戶集區造成百分比影響。

### <a name="define-a-solution-to-test-the-hypothesis"></a>定義解決方案以測試假設

在 build-measure-學習意見反應迴圈的每個反復專案中，您嘗試以理解方式建立時，是由 MVP 所定義。

MVP 是 (發明、工程、應用程式開發或資料結構所需的最小單位，) 建立足以向 *客戶*學習的解決方案。 每位 MVP 的目標是要測試部分或所有先前的假設，並直接從客戶收到意見反應。 輸出不是美觀的應用程式，具有變更產業所需的所有功能。 每個反復專案的所需輸出是學習機會，有機會更深入測試假設。

*Timeboxing* 是確保產品保持精簡的標準方式。 例如，請確定您的開發小組認為解決方案可以在單一反復專案中建立，以允許快速測試。 若要進一步瞭解使用速度、反復專案和版本來定義最小的意義，請參閱 [規劃速度、反復專案、發行和反復專案路徑](../../plan/iteration-paths.md)。

### <a name="reduce-complexity-and-delay-technical-spikes"></a>降低複雜性並延遲技術尖峰

在「[創新」方法](./index.md)中找到的[發明專業領域](./invention.md)，會說明傳遞成熟的創新或已就緒的 MVP 解決方案所需的功能。 使用這些專業領域作為功能納入的長期指南。 同樣地，您也能在及早測試客戶價值和解決解決方案的過程中，使用它們作為警告性指南。

功能廣度和發明的不同專業領域無法在單一反復專案中建立。 MVP 解決方案可能需要數個版本，以包含多個專業領域的複雜度。 根據開發的投資，可能會有多個平行團隊在不同的專業領域內運作，以測試多個假設。 雖然可以在這些小組之間保持架構的一致性，但造成嘗試建立複雜的整合式解決方案，直到可以驗證值假設為止。

最適合在 *技術尖峰*的頻率或數量中偵測到複雜度。 技術尖峰是為了建立無法輕鬆測試客戶的技術解決方案。 當客戶價值和客戶理解未經過測試時，技術尖峰就代表創新的風險，應該將其最小化。 針對在遷移工作中發現的成熟測試解決方案，技術尖峰可能會在採用期間很普遍。 不過，它們會延遲測試假設的測試，而且應該盡可能延後。

針對任何 MVP 定義，建議使用一種非常簡化的方法。 這種方法表示移除任何未加入您驗證假設的功能。 若要將複雜性降至最低，請減少測試假設所需的整合和功能數目。

### <a name="build-an-mvp"></a>打造 MVP

在每個反復專案中，MVP 解決方案可以採用許多不同的圖形。 常見的需求是，輸出可進行假設的測量和測試。 這項簡單的需求會起始科學流程，並讓小組以理解的方式打造。 為了提供此客戶優先的焦點，初始 MVP 可能只依賴發明的其中一個 [專業領域](./invention.md)。

在某些情況下，創新的最快途徑表示暫時避免這些專業領域，直到雲端採用小組確信已正確驗證假設為止。 來自 Microsoft 這類技術公司，本指南可能會違反直覺。 但是，這只是為了強調客戶的需求，而不是特定的技術決策，是 MVP 解決方案中最高的優先順序。

MVP 解決方案通常是由簡單的應用程式或資料解決方案所組成，其中包含最少的功能和有限的波蘭文。 對於具有專業開發專業知識的組織來說，此路徑通常是最快學習和反復專案的途徑。 下列清單包含小組建立 MVP 時可能採取的其他數種方法：

- 一種預測演算法，此演算法的99% 時間錯誤，但會示範特定的預期結果。
- IoT 裝置不會在生產環境中安全地進行通訊，而是示範進程內幾乎即時資料的價值。
- 公民開發人員所建立的應用程式，用來測試假設或符合較小規模的需求。
- 重新建立應用程式所需遵循之優點的手動程式。
- 詳細說明的線框或影片，可讓客戶進行互動。

開發 MVP 不需要大量的開發投資。 建議您最好盡可能地限制投資，以將一次測試的假設數目降到最低。 然後，在每個反復專案及每個版本中，都會刻意針對表示多個發明專業領域的擴充性解決方案來改善解決方案。

### <a name="accelerate-mvp-development"></a>加速 MVP 開發

上市時間對於任何創新的成功都很重要。 更快速的發行可加快學習速度。 更快速的學習會導致能更快擴充的產品。 有時候，傳統的應用程式開發週期可能會使此程式變慢。 更頻繁地來說，創新受限於可用的專業知識。 員工的預算、員工和可用性都可以對小組可以處理的新創新數量產生限制。

人力限制和想要以理解的方式打造，為公民開發人員產生快速成長的趨勢。 這些開發人員可降低風險，並在組織的專業開發小組內提供規模。 公民開發人員是客戶經驗所關注的主題專家，但他們並不是以工程師的形式定型。 這些個人使用原型工具或較輕量的開發工具，這些工具可能由專業開發人員 frowned。 這些與商務一致的開發人員會建立 MVP 解決方案和測試理論。 當對齊時，此程式可以建立提供價值的生產解決方案，但不會傳遞夠有效的規模假設。 它們也可以用來在開始調整規模之前驗證原型。

在任何創新方案中，雲端採用小組應 [多樣化其組合，以包含公民開發人員工作。 藉由調整開發工作，可在較低的投資中形成更多假設並進行測試。 當某個假設經過驗證併發現可定址的市場時，專業開發人員就可以使用新式開發工具來強化及調整解決方案。

### <a name="final-build-gate-customer-pain"></a>最終組建閘道：客戶難題

當客戶理解的是強式時，顯然存在的問題應該很容易識別。 客戶的難題應該很明顯。 在組建期間，雲端採用小組正在建立解決方案，以根據客戶痛點測試假設。 如果假設是妥善定義的，但痛點不是，則解決方案並非真正根據客戶的理解。 在此案例中，build 不是正確的起點。 相反地，請先投資，並從真實客戶學習並學習。 建立理解與驗證難題的最佳方法很簡單：聆聽您的客戶。 花點時間開會並觀察它們，直到您能找出經常發生的難題。 當您清楚瞭解問題之後，就可以開始測試假設解決方案來解決這個難題。

## <a name="when-not-to-apply-this-approach"></a>不套用此方法的時機

許多需要採用替代方法的法律、合規性和產業需求。 如果開發解決方案的公開版本建立了專利時間、智慧財產權保護、客戶資料流程失或違反既定合規性需求的風險，則這種方法可能不適用。 如果發現這類的風險存在，請先洽詢法律顧問，再採用任何發行管理的引導式方式。
