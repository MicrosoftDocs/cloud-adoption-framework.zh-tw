# <a name="define-a-security-strategy"></a>定義安全性策略

安全性組織的最終目標並不會隨著雲端服務的採用而改變，但這些目標的達成方式將會改變。 安全性小組仍然必須專注于降低遭受攻擊的商業風險，以及取得內建于所有資訊系統和資料的機密性、完整性和可用性保證。

## <a name="modernize-your-security-strategy"></a>現代化您的安全性策略

安全性小組必須將策略、架構和技術現代化，因為組織會採用雲端，並隨著時間運作。 雖然變更的大小和數目一開始可能令人望而生畏，但安全性計畫的現代化可讓安全性縮減一些與舊版方法相關的棘手負擔。 組織可能會暫時使用舊版策略和工具，但這種方法很容易因雲端和威脅環境中的變更步調而承受：

- 如果安全性小組採用「arm-長度」安全性的傳統思維方式，其中的答案一律以「否」開頭（而不是與 IT 和商務團隊一起合作，以降低風險，而使企業變得更低），則可能會離開雲端採用決策。
- 如果安全性小組僅使用舊版的內部部署工具，而且僅遵守網路周邊 doctrine 來進行所有防禦和監視，則會在偵測和防禦雲端攻擊時遇到艱難的時間。 以雲端規模來保護會要求使用雲端原生偵測和自動化功能，以及引進身分識別周邊，以協助監視及保護雲端和行動資產。

因為這項轉換可能很重要，所以我們建議安全性小組採用敏捷式方法來現代化安全性，以快速將其現代化策略的最重要層面，然後在該時間之後持續累加地進行改善。

### <a name="security-of-the-cloud-and-from-the-cloud"></a>雲端和雲端的安全性

當您的組織採用雲端服務時，安全性小組將會努力達到兩個主要目標：

- **雲端的安全性 \* \* （保護雲端資源）：** 安全性應該整合到雲端服務的規劃和操作，以確保這些核心安全性保證會一致地套用到所有資源上。
- ** \* \* 雲端的安全性（使用雲端來轉換安全性）：** 安全性應該立即開始規劃及思考如何使用雲端技術將安全性工具和程式現代化，特別是原生整合的安全性工具。 越來越多的安全性工具裝載于雲端，並提供難以或無法在內部部署環境中執行的功能。

許多組織一開始會將雲端資源視為額外的_虛擬資料中心_，其運作方式非常類似雲端安全性的起點。 當組織使用雲端的安全性現代化時，大部分的想法都會快速過度擴增此模型。 使用雲端裝載的工具保護軟體定義的資料中心，可讓您超越內部部署模型所能提供的功能：

- 快速啟用和調整安全性功能。
- 高效率的資產清查和安全性設定防護探索。
- 持續評估組織的安全性狀態和控制項。
- 大幅改善威脅偵測，利用龐大的威脅情報存放庫，以及幾乎不受限制的雲端處理/儲存。

### <a name="the-right-level-of-security-friction"></a>正確的安全性摩擦層級

安全性自然建立了減緩進程的摩擦，識別哪些元素在您的 DevOps 和 IT 程式中是狀況良好的，而哪些不是：

- **狀況良好的摩擦：** 就像是在練習中的抵抗，讓肌肉更強，整合適當的安全性摩擦，藉由在適當的時間強制執行重要的思考，來增強系統或應用程式。 這種方式通常會考慮到攻擊者在設計期間如何嘗試危害應用程式或系統，以及如何在軟體程式碼、設定或操作實務中，檢查、識別和理想地修正攻擊者可以利用的潛在弱點。
- **狀況不良的摩擦：** 會妨礙超出其保護的價值。 這種情況通常發生在工具產生的安全性錯誤（例如錯誤警示）時，或當探索或修正安全性問題的麻煩超過攻擊的潛在影響時。

### <a name="standalone-and-integrated-responsibilities"></a>獨立和整合責任

若要提供機密性、完整性和可用性保證，安全性專家必須負責操作專用的安全性功能，以及與組織中的其他小組密切合作：

- **唯一的安全性功能：** 安全性小組會執行獨立功能，在組織中的其他地方找不到，例如安全性作業、弱點管理和其他功能。
- 將**安全性整合至其他**函式：安全性小組也可以做為組織中其他小組和職能的主題專家，這些團隊和功能會推動商務計畫、評估風險、設計或開發應用程式，以及操作 IT 系統。 安全性小組會向這些團隊提供攻擊者的專業知識和內容、攻擊方法和趨勢、可能允許未經授權存取的弱點，以及風險降低步驟或因應措施及其潛在優點或陷阱的選項。 這項安全性功能與品質函式類似，因為它會在多個位置中結合，以支援單一結果。

在這些責任上執行，同時掌握雲端的快速變更步調和企業轉型，將需要安全性小組將他們的工具、技術和流程現代化。

## <a name="transformations-mindsets-and-expectations"></a>轉換、心態和預期

許多組織會管理組織中多個同時轉換的鏈。 這些內部轉換通常會開始，因為幾乎所有的外部市場都會轉換，以符合行動和雲端技術的新客戶喜好設定。 組織通常面臨新新創公司的競爭威脅，以及可能中斷市場之傳統競爭者的數位轉型。

![組織中多個同時轉換的鏈](../_images/security/transformation-chain.png)

內部轉換程式通常包括：

- 企業的**數位轉型**，為您創造新的商機，並與數位原生新創公司保持競爭力。
- IT 組織的**技術轉型**，以支援雲端服務、現代化開發實務和相關變更的計畫。
- **安全性轉換**可同時適應雲端，並同時滿足日益複雜的威脅環境。

### <a name="internal-conflict-can-be-costly"></a>內部衝突可能會很耗費成本

變更會建立壓力和衝突，這可能會方格終止的決策。 這對於安全性風險責任通常會在主題專家（安全性小組）上錯置，而不是在負責商業結果和所有其他風險類型的資產（企業擁有者）的擁有者上，特別是如此。 這種錯誤的責任通常是因為所有的專案關係人不正確地將安全性視為要解決的技術或絕對問題，而不是像是公司間諜和其他傳統刑事活動等動態持續性的風險。

在這段轉換期間，所有團隊的領導都必須積極努力，以減少疏失重要專案和 incentivize 團隊的衝突，以略過安全性風險降低。 小組之間的 Internecine 衝突可能會導致：

- **增加安全性風險**，例如肇因安全性事件或增加攻擊的業務損害（特別是當團隊因安全性而感到挫折，或是攻擊者輕易略過過時的安全性方法時）。
- 對**企業或任務的負面影響**，例如當商務程式未啟用或更新為符合市場需求時（通常是在安全性程式保有重要的商務計畫時）。

請務必留意小組內部和之間的關聯性健全狀況，以協助他們導覽改變的橫向，讓有價值的小組成員不安全和未結算。 這些心態的耐心、學習和教育以及未來的正面可能會協助您的小組更清楚地流覽這段期間，為組織提供良好的安全性結果。

領導人可以利用具體的主動式步驟來協助進行文化特性變更，例如：

- 公開模型化其小組預期的行為。
- 對變更的挑戰而言是透明的，包括反白顯示他們自己的掙扎來調整。
- 定期提醒小組現代化和整合安全性的緊急性與重要性。

### <a name="cybersecurity-resilience"></a>網路安全性復原

許多傳統的安全性策略僅著重于預防攻擊，這對新式威脅而言並不夠。 安全性小組必須確保其策略超出此範圍，同時也能提供快速的攻擊偵測、回應和復原，以提高恢復能力。 組織必須假設攻擊者會危害一些資源（有時候稱為「假想缺口」），並確保資源和技術設計在攻擊防護與攻擊管理之間達到平衡（而不是僅嘗試防止攻擊的一般預設方法）。

許多組織已經在此旅程上，因為他們一直在過去幾年來管理穩定增加的大量和複雜的攻擊。 此旅程通常是從第一個主要事件開始，這可能是一項情緒事件，讓人們失去其先前的 invulnerability 和安全性。 雖然這種情況並不嚴重，但此事件可能會觸發類似的表情，從拒絕開始，最後在接受時結束。 這種「失敗」的假設一開始可能會很難接受，但它與妥善建立的「不安全」工程原則有強烈的關係，而且假設可讓您的小組專注于更佳的成功定義：恢復功能。

[NIST 網路安全性架構](https://www.nist.gov/cyberframework)的功能是一項實用的指南，說明如何在可復原的策略中識別、保護、偵測、回應和復原的互補活動之間，平衡投資。

如需網路安全性復原的詳細資訊，以及網路安全性控制項的最終目標，[請見如何讓組織的風險降低](https://docs.microsoft.com/azure/architecture/framework/security/resilience)。

### <a name="how-the-cloud-is-changing-security"></a>雲端如何變更安全性

轉移到雲端以獲得安全性，不只是一項簡單的技術變更，而是從大型主機移到桌上型電腦，再到企業伺服器上的技術一代轉移。 成功流覽這項變更需要安全性小組對期望和思維的基礎轉移。 採用適當的心態和預期會降低組織內的衝突，並提升安全性小組的效益。

雖然這些可能是任何安全性現代化計畫的一部分，但雲端中的快速變更速度會使其採用緊急的優先順序。

- **與共享目標的合作關係。** 在這段快速的決策和持續的程式演進方面，安全性無法再採用「arm 長度」方法來核准或拒絕對環境所做的變更。 安全性小組必須與企業和 IT 小組密切合作，以建立有關生產力、可靠性和安全性的共用目標，並與這些合作夥伴共同合作來達成這些目的。

  這項合作關係是「向左移位」的最終形式，這是在程式 &mdash; 中整合安全性的原則，讓修正安全性問題變得更容易且更有效率。 這需要所有相關的文化特性變更（安全性、商業和 IT），要求每一個都要學習文化特性和其他群組的規範，同時同時教其他人自己的意見。

  安全性小組必須：

  - **學習**企業和 IT 目標，以及每個專案的重要性，以及他們在進行轉換時的思考方式。
  - **分享**為什麼安全性在這些商業目標和風險的內容中很重要，還有其他小組可以做什麼來達成安全性目標，以及他們該怎麼做。

  雖然這不是簡單的工作，持續保護組織及其資產是不可或缺的。 這項合作關係可能會導致健全的折衷，一開始只會符合最低安全性、商務和可靠性目標，但一段時間後又逐漸改善。

- **安全性是一種持續的風險，而不是問題。** 您無法「解決」犯罪。 就它的核心而言，安全性只是風險管理專業領域，而不是自然的事件，而是專注于惡意動作。 就像所有風險一樣，安全性並不是可以由解決方案修正的問題，而是從負面事件、攻擊的可能性和影響的組合。 它最適合傳統的公司間諜和刑事活動，組織會面對有權進行財務獎勵，以成功攻擊組織的人。

- **無論是生產力或安全性，都需要兩者都成功。** 組織必須專注于現今「創新或變得無關」環境中的安全性與生產力。 如果組織沒有產能並推動新的創新，可能會失去 marketplace 中的競爭力，使其無法進行財務或最終失敗。 如果組織不安全，而且無法控制對攻擊者的資產，則可能會失去 marketplace 中的競爭力，使其無法進行財務並最終失敗。

- **沒有人完美。** 不適合採用雲端，甚至是 Microsoft。 Microsoft 的 IT 和安全性團隊大家常掙扎了許多與我們的客戶一樣的挑戰，像是找出如何妥善處理程式、平衡支援的舊版軟體，以及支援先進的創新，甚至是雲端服務的技術缺口。 當這些小組學習如何更有效地操作和保護雲端時，他們會主動透過這類檔與[IT 展示網站](https://www.microsoft.com/itshowcase)上的其他人分享其經驗，同時持續提供意見反應給我們的工程小組和協力廠商，以改善其供應專案。

  根據我們的經驗，我們建議小組保留為持續學習和改進的標準，而不是完美的標準。

- **轉換的機會。** 請務必將數位轉型視為安全性的正面商機。 雖然您很容易看到這項變更的潛在缺點和風險，但是您可以輕鬆地錯過大幅提高安全性角色的機會，並在資料表上取得決策。 與企業合作可以提高安全性資金、降低安全性的重複性工作，並讓您更輕鬆地使用安全性，因為他們將更能連接到組織的任務。

## <a name="adopting-the-shared-responsibility-model"></a>採用共用責任模型

在雲端中裝載 IT 服務，可在雲端提供者與客戶租使用者之間分割工作負載的營運和安全性責任，建立與共同責任的實際合作關係。 所有安全性小組都必須研究並瞭解此共同責任模型，才能將其程式、工具和技能集調整為新世界。 這有助於避免不小心在安全性狀態中建立間距或重迭，因而導致安全性風險或資源浪費。

此圖說明如何在雲端廠商與雲端客戶組織之間散佈安全性責任，而這是一項事實上的合作關係：

![分散式安全性責任](../_images/security/security-responsibilities.png)

由於雲端服務有不同的模型，每個工作負載的責任會根據其裝載于軟體即服務（SaaS）、平臺即服務（PaaS）、基礎結構即服務（IaaS）或內部部署資料中心而有所不同。

## <a name="building-security-initiatives"></a>建立安全性計畫

此圖說明大部分安全性程式應該遵循的三個主要安全性計畫，以調整其在雲端的安全性策略和安全性計畫目標：

![主要的安全性計畫](../_images/security/primary-security-initiatives.png)

在雲端中建立復原的安全性狀態需要數個平行互補的方法：

- **信任但驗證：** 針對雲端提供者所執行的責任，組織應採取「信任但驗證」的方式。 組織應評估其雲端提供者的安全性作法，以及他們所提供的安全性控制，以確保雲端提供者符合組織的安全性需求。

- **現代化基礎結構和應用程式安全性：** 對於組織控制之下的技術專案，請設定現代化安全性工具的優先順序，並建立技能集的關聯性，以將雲端中資源的保護措施降至最低。 這是由兩個不同的互補工作所組成：

  - **基礎結構安全性：** 組織應使用雲端來現代化其保護和監視許多應用程式所使用之萬用群組件（例如作業系統、網路和容器基礎結構）的方法。 這些雲端功能通常會包括管理 IaaS 和內部部署環境中的基礎結構元件。 將此策略優化很重要，因為此基礎結構是在其上執行之應用程式和資料的相依性，這通常會啟用重要的商務流程並儲存重要的商務資料。
  - **應用程式安全性：** 組織也應該將其組織所開發的獨特應用程式和技術的保護方式現代化。 這個專業領域會隨著 agile DevOps 流程的採用、增加開放原始碼元件的使用，以及引進雲端 Api 和雲端服務來取代應用程式元件或互連應用程式，而快速改變。

    取得這個許可權非常重要，因為這些應用程式通常會啟用重要的商務程式並儲存重要的商務資料。

  - **現代化周邊：** 組織應具備完整的方法來保護所有工作負載的資料，組織應建立一致、集中管理的身分識別控制項的現代化周邊，以保護其資料、裝置和帳戶。 這會受到在 CISO 研討會的模組3中詳細討論的零信任策略，以及更多資源的影響。

### <a name="security-and-trust"></a>安全性與信任

請注意，在安全性中使用「信任」一詞可能會造成混淆。 這份檔會以兩種方式來說明這種概念的實用應用程式：

- 「[零信任](https://www.microsoft.com/security/business/zero-trust)」是一種常見的業界術語，適用于安全性的策略方法，假設公司或內部網路都是惡意的（值得一提的「零信任」）並據此設計安全性。
- [信任但驗證](https://en.wikipedia.org/wiki/trust,_but_verify)是一種運算式，它會將兩個不同組織的本質與共同作業共同運作，儘管有一些其他可能會造成分歧的興趣。 這可讓您在與公司的商業雲端提供者合作的早期階段中，簡化許多細微的細節。

雲端提供者及其實務和流程可能會負責符合合約和法規需求，而且可能會贏得或失去信任。 網路是一種 nonliving 連線，如果攻擊者使用此連線，就不會面臨後果（就像您無法持有使用他們的犯罪或汽車）。

## <a name="how-cloud-is-changing-security-relationships-and-responsibilities"></a>雲端如何改變安全性關係和責任

如同先前的轉換成新一代的技術（例如桌上型電腦運算和企業伺服器運算），轉移到雲端運算會中斷長期建立的關聯性、角色、責任和技能集。 過去幾年來，我們習慣的工作描述並不完全對應到現在包含雲端功能的企業。 隨著產業共同致力於將新模型正規化，組織就必須專注于盡可能地提供更清楚的資訊，以協助管理這段變更期間不明確的不確定性。

安全性小組會受到這些變更的影響，也就是他們所支援的商業和技術，以及他們自己的內部現代化工作，為威脅執行者提供更好的方向。 攻擊者不斷演進，持續搜尋組織人員、程式和技術的最簡單弱點，而且安全性必須開發能力和技能來滿足這些角度。

本節說明在雲端旅程上經常變更的主要關聯性，包括將風險降至最低和促進機會改善的課程：

- **在安全性與商務專案關係人之間：** 安全性領導能力必須與業務領導人協力合作，才能讓組織降低風險。 安全性領導人應支援以安全性主題專家（Sme）為目標的商業決策，並應致力於將受信任的顧問成長到這些商業領導人。 這項關聯性將有助於確保商務領導人在做出商業決策、通知商業優先順序的安全性，並協助確保安全性投資與其他投資適當地設定優先順序。

- **在安全性領導和團隊成員之間：** 安全性領導地位應將這些深入解析的商業領導權帶回其小組，以引導其投資優先順序。

  藉由設定與業務領導人和其小組的合作，而不是傳統的「arm 長度」關聯性，安全性領導人可以避免形成對立動態，同時妨礙安全性和生產力目標。

  安全性領導人應努力為其小組提供清楚的方式，以瞭解如何管理他們對於生產力和安全性取捨的每日決策，因為這可能會對其小組而言很新。

- **在應用程式和基礎結構小組（和雲端提供者）之間：** 這項關聯性正在進行重大變更，因為 IT 和安全性產業中有多個趨勢，其目標是要增加創新速度和開發人員生產力。

  舊的規範和組織函式已中斷，但新的規範和函式仍在推出中，因此我們建議您接受不明確的事項、掌握目前的想法，以及試驗貴組織的運作方式。 我們不建議在此空間中採用等待和查看方法，因為它可能會讓您的組織面臨主要的競爭優勢。

  這些趨勢對應用程式和基礎結構的角色和關聯性的傳統規範具有挑戰性：

  - **DevOps-融合專業領域：** 在它的理想狀態下，這會有效率地建立單一的高功能小組，將這兩組主題相關的專業知識結合在一起，以快速創新、發行更新並解決問題（安全性，否則為）。 雖然這種理想的狀態需要一些時間才能達成，而且中間的責任仍然非常不明確，但由於這種合作方式，所以組織已經享受了快速發行的一些優點。 Microsoft 建議您將安全性整合到此週期，以協助瞭解這些文化特性、分享安全性學習，以及處理快速發行安全且可靠之應用程式的常見目標。

  ![DevOps-融合專業領域](../_images/security/devops-disciplines.png)

  - **容器化成為一個通用的基礎結構元件：** 應用程式逐漸受到 Docker、Kubernetes 和類似技術等技術的裝載和協調。 這些技術藉由抽象化基礎作業系統的安裝和設定的許多元素，來簡化開發和發行作業。

  ![容器化基礎結構](../_images/security/containerization.png)

  雖然容器是以開發小組所管理的應用程式開發技術開始進行，但它會成為日益轉移到基礎結構小組的通用基礎結構元件。 這項轉換在許多組織中仍在進行中，但它是自然且正向的方向，使用傳統的基礎結構技能集（例如網路、儲存體和容量管理），就能更有效地解決目前的各種挑戰。

  支援的基礎結構小組和安全性小組成員應提供訓練、程式和工具，協助您管理、監視和保護這項技術。

  - **無伺服器和雲端應用程式服務：** 業界的主要趨勢之一，就是減少建立或更新應用程式所需的時間和開發工作。

    ![無伺服器和雲端應用程式服務](../_images/security/serverless-model.png)

    開發人員也逐漸使用雲端服務來執行下列動作：

    - 執行程式碼，而不是在虛擬機器（Vm）和伺服器上裝載應用程式。
    - 提供應用程式功能，而不是開發自己的元件。 這會導致使用現有雲端服務進行一般功能的_無伺服器_模型。 雲端服務的數目和種類（以及其創新步調）也已超過安全性小組評估和核准使用這些服務的能力，讓他們可以選擇讓開發人員使用任何服務、嘗試防止開發小組使用未經核准的服務，或試著找出更好的方法。
    - **無程式碼應用程式和電源應用程式：** 另一個新興的趨勢是使用無程式碼技術（例如 Microsoft power apps）。 這項技術可讓不具編碼技能的人員建立可達成商業成果的應用程式。 由於這項低摩擦和高價值的潛能，這項趨勢可能會快速增加普及，而安全性專業人員可以快速瞭解其含意。 安全性工作應著重于人類可能在應用程式中犯錯誤的領域，也就是應用程式和資產許可權的設計，透過威脅模型化應用程式元件、互動/關聯性和角色許可權。

- **開發人員和開放原始碼元件作者：** 開發人員也可以使用開放原始碼元件和程式庫來提高效率，而不是開發自己的元件。 這會透過效率帶來價值，但也會藉由建立外部相依性以及適當維護和修補這些元件的需求，來引進安全性風險。 開發人員在使用這些元件時，會有效地假設安全性和其他 bug 的風險，而且必須確保計畫以與他們所開發的程式碼相同的標準來減輕這些問題。

- **在應用程式與資料之間：** 資料和應用程式的安全性這一行會變得模糊，而新的法規則是在資料/隱私權團隊和安全性小組之間建立更密切的合作需求：

  - **機器學習（機器學習）演算法：** 機器學習服務演算法與應用程式相似之處在于，它們是設計來處理資料來建立結果。 主要差異包括：

    - **高價值機器學習服務：** 機器學習服務通常會授明顯的競爭優勢，而且通常會被視為機密的智慧財產權和貿易秘密。

    - **敏感性：** 受監督的機器學習會使用資料集進行微調，這會在演算法上 imprints 資料集的特性。 因此，微調的演算法可能會被視為機密，因為資料集會用來進行定型。 例如，使用秘密軍隊基底的資料集，將機器學習演算法定型以尋找地圖上的秘密軍隊基底，會使其成為敏感性資產。

    > [!NOTE]
    > 並非所有範例都很明顯，因此請務必將小組與資料科學小組、商務專案關係人、安全性小組、隱私權小組及其他專案的正確關係人結合在一起。 這些小組應負責滿足創新和責任的共同目標。 他們應該解決常見的問題，例如如何及何處將資料的複本儲存在不安全的設定、如何分類演算法，以及您組織的任何疑慮。
    >
    > Microsoft 已發佈[負責的 AI 原則](https://www.microsoft.com/ai/responsible-ai)，以引導我們自己的小組和我們的客戶。

    - **資料擁有權和隱私權：** GDPR 之類的法規已增加資料問題和應用程式的可見度。 應用程式小組現在可以控制、保護和追蹤層級的敏感性資料，其可與依銀行和金融機構追蹤財務資料的程度相同。 資料擁有者和應用程式小組必須針對資料應用程式的儲存和需要哪些控制項，建立豐富的瞭解。

- **在組織與雲端提供者之間：** 當組織在雲端裝載工作負載時，他們會與每個雲端提供者進入商務關係。 使用雲端服務通常會供應商業價值，例如：

  - 藉由減少新功能的上市時間，**加速數位轉型計畫**。

  - 藉由釋出團隊以專注于較高價值（商務對齊）活動，而不是由雲端服務代表其提供更有效率的較低層級商用工作，**提高 IT 和安全性活動的價值**。

  - **更高的可靠性和回應性：** 相較于傳統內部部署資料中心，大多數新式雲端也有極高的執行時間，並已顯示它們可以快速調整（例如在 COVID-19 流感期間），並提供下列自然事件的復原功能（這會讓多個內部部署的對等專案變得更長）。

    雖然非常有説明，但對雲端的轉移並不會有任何風險。 當組織採用雲端服務時，他們應該考慮潛在的風險區域，包括：

  - **商務持續性和嚴重損壞修復：** 雲端提供者是否會在公司使用服務的過程中，針對可能會存活的商務模型和在其上發展的業務模式狀況良好？ 如果提供者遇到財務或其他失敗（例如，將其原始程式碼提供給客戶或開放原始碼），雲端提供者會布建以允許客戶持續性嗎？

    如需有關 Microsoft 財務健全狀況的詳細資訊和檔，請參閱[microsoft 投資者關係](https://www.microsoft.com/investor)。

  - **安全性：** 雲端提供者是否遵循業界的安全性最佳作法？ 這是由獨立的法規主體驗證的嗎？

    - [Microsoft cloud app security](https://docs.microsoft.com/cloud-app-security/risk-score)可讓您探索超過16000個雲端應用程式的使用方式，其會根據70個以上的風險因素進行排名和評分，讓您持續掌握雲端使用方式、影子 it 以及影子 it 對組織造成的風險。
    - [Microsoft 服務信任入口網站](https://servicetrust.microsoft.com)可讓客戶取得法規合規性認證、審核報告、畫筆測試和更多功能。 這些檔包含內部安全性實務的許多詳細資料（特別是 SOC 2 第2類報告和 FedRAMP 中等系統安全性計畫）。

  - **商務競爭者：** 雲端提供者是您產業中的重要商務競爭者嗎？ 您是否有足夠的雲端服務合約保護，或其他方法來保護您的企業免于可能的惡意動作？

    請參閱[這篇文章](https://www.cnbc.com/2019/03/16/why-volkswagen-chose-microsoft-azure-over-aws.html)，以瞭解 Microsoft 如何避免與雲端客戶競爭。

  - **多重雲端：** 許多組織都有刻意或刻意的多重雲端策略。 這可能是一個刻意的目標，以降低對單一供應商的依賴或存取獨特的最佳功能，但也可能是因為開發人員選擇偏好或熟悉的雲端服務，或是您的組織取得了另一家企業。 不論原因為何，此策略可能會造成潛在的風險，以及必須管理的成本，包括：

    - **來自多個相依性的停機時間：** 架構為依賴多個雲端的系統會暴露于更多的停機時間來源，因為雲端提供者（或您的小組使用他們）的中斷，可能會導致您的業務中斷/中斷。 這種增加的系統複雜度也會增加中斷事件的可能性，因為小組成員比較不可能完全瞭解較複雜的系統。
    - **談判動力：** 較大的組織也應該考慮單一雲端（相互承諾/合作關係）或多重雲端策略（轉移業務的能力）是否會對其雲端提供者產生更大的影響，以排定其組織的功能要求。
    - **增加維護額外負荷：** IT 和安全性資源已經從現有的工作負載中進行過多工作，並與單一雲端平臺的變更保持一致。 每個額外的平臺會進一步增加這項負擔，並使小組成員遠離較高價值的活動，例如簡化技術程式以加速商業創新、針對更有效率的技術使用業務小組諮詢等等。
    - **人員配置和訓練：** 組織通常不會考慮支援多個平臺所需的人員配置需求，以及維護以快速步調發行之新功能的知識和貨幣所需的訓練。
