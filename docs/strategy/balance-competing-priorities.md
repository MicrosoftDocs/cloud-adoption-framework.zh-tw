# <a name="balance-competing-priorities"></a>平衡競爭優先順序

在任何數位轉型旅程上登機的行為，就像專案關係人在商務和技術團隊的強制函式一樣。 成功的路徑是以組織能夠平衡競爭優先順序的能力為基礎進行穩固的。

類似于其他數位轉換，雲端採用會在採用生命週期中公開競爭優先順序。 就像其他形式的轉換一樣，在這些優先權中找出餘額的能力，將會對商業價值的實現有顯著的影響。 若要平衡這些競爭優先順序，必須在專案關係人（有時與個別參與者）之間進行開放式和有時不容易的交談。

本文概述在每個方法執行期間經常討論的一些競爭優先順序。 我們希望這個先進的認知有助於在開發雲端採用策略時，為這些討論做更好的準備。

## <a name="general-theme-of-the-cloud-adoption-framework-approach"></a>雲端採用架構方法的一般主題

整合型解決方案和先進規劃都是建置於一系列假設上，可能（或可能不會）證明在一段時間內是精確的。 採用雲端通常是企業和技術團隊的新體驗。 就像大多數的新體驗或學習機會一樣，這些假設會有很高的機率會證明 false。

遵循經過證實的延遲技術決策 agile 原則，是此架構中大部分指引的最優先方法。 該方法遵循一致的模式：建立一般的端點狀態、快速地移至初始執行、測試和驗證假設，並及早重構以解決假設。 這種類型的成長思維可將學習最大化，並將風險降至最低，但需要一些明確的緩和。

有時候，可能會 scarier （或更危險）不明確的假設。 雖然此架構仰賴在執行期間學習和解決不明確的情況，但是在許多情況下，小組必須針對以分析或假設為基礎的方法進行深入瞭解。 下列各節將嘗試說明每個區段中至少一個「展開的範圍範例」，以說明第二個更深入的反復專案有價值時的時間。

## <a name="balance-during-the-strategy-phase"></a>策略階段期間的餘額

策略方法的核心目標是要開發專案關係人之間的對齊。 一旦定義之後，符合的策略性位置就會在每個方法中推動行為，以確保技術決策符合所需的商務結果。 專案關係人之間的培養對齊會建立一組常見的競爭優先順序：相對**于業務影響****的深度**與時間之間的差異。

**競爭優先順序：**

- **理由的深度：** 專案關係人通常會想要進行深入的財務分析和完整的商業理由，以配合策略性的方向。 可惜的是，該層級的分析可能需要延長的時間週期，以允許資料收集和分析。
- **時間對企業的影響：** 相反地，專案關係人通常會負責在定義的時間範圍內傳遞商務結果。 耗時的分析和評量可以在技術工作開始之前，將這些結果放在風險中。

**最小範圍：** 尋找此餘額需要及早在程式中進行專案關係人討論。 策略方法會建議在此初期工作中限制對齊的範圍。 在建議的方法中，專案關係人著重在一組核心動機、可測量的結果，以及高層級的商業理由。 然後，專案關係人應該可以快速地認可到少量的初始專案或試驗，以推動必要的學習機會。

**擴充的範圍範例：** 如果最初的商務分析指出對業務造成負面影響的高風險，則專案關係人可能需要減緩並更謹慎地評估商業論證期間更深入的分析。

## <a name="balance-during-the-plan-phase"></a>計畫階段期間的餘額

與策略階段優先順序類似，在計畫階段中，需要平衡初始規劃與延遲技術決策的深度。

**競爭優先順序：**

- 關於雲端中的技術實作為**初步規劃的深度，** 通常包含大量的假設。 特別是當小組有技能差距時，環境會受到探索缺口的影響，或工作負載沒有清楚定義的架構結束狀態。 這些假設全都在詳細的雲端採用計畫中很常見。 需要進行實驗、試驗和定性分析，才能移除這些假設。
- **延遲的技術決策**假設您稍後可以進行技術決策，這會是更精確的決策。 遵循 agile 產品規劃的原則將有助於延遲技術決策，讓他們能夠在適當的時間內使用足夠的資訊。 不過，這種方法會在初始計畫中產生更高程度的不明確。

**最小範圍：** 建議靈活的產品開發方法，以在可管理的計畫內進行「磁片磁碟機提示」動作。 計畫方法會建議下列步驟來達成這項平衡。 使用自動化的探索工具清查完整的數位資產，但使用累加式合理化來規劃未來1-3 個月的工作。 確保適當的組織對齊以快速移動。 為指派的小組建立技能的準備就緒計畫。 使用 [雲端採用方案] 範本來快速部署初始待處理專案。

**擴充的範圍範例：** 有時候，雲端採用方案的傳遞可能會回應時間緊迫或高影響力的商務活動。 當成功需要在一段固定時間內移動大量資產時，上述步驟通常會遵循更深入的規劃工作。 在這些案例中，成功的關鍵在於規劃是否足以開始進行，然後規劃完整的參與。 這種方法可降低規劃封鎖商務結果的可能性。

## <a name="balance-during-the-ready-phase"></a>準備階段期間的餘額

當採用小組準備將其第一個步驟加入雲端時，採用和長期作業的時間通常會有競爭的優先順序。 小組可能會擔心非常適合在手邊提供工作，而不是妥善管理。 這在傳統 IT 環境中是必要的，而開發平臺的動作需要實體資產和收購週期。 不過，在程式碼中定義整個 IT 平臺時，傳統的開發策略（例如重構）可減少從一開始就妥善管理的需求。

**競爭優先順序：**

- **長期作業：** 客戶通常會遭到封鎖，讓雲端環境符合目前作業管理、治理和安全性系統的功能同位。 在目前的客戶研究中，超過90% 的客戶需要支援超過這項思維。 此封鎖程式會插入延遲變慢或防止業務衝擊的月份。
- **採用時間：** Azure 原則、Azure 藍圖和管理群組等雲端式工具可讓您輕鬆地跨 IT 平臺進行重構。 此外，預先定義的登陸區域會提供固定的位置，以加速部署至已經符合許多功能同位需求的環境。 同時也有機會加速上市時間，對長期營運的影響最小。

**最小範圍：** Ready 方法概述從快速採用到長期作業的直接路徑。 這個方法是從啟用環境重構的工具的基本簡介開始。 根據這些工具和環境需求，會引導客戶選擇預先定義的登陸區域（每個都是使用基礎結構即程式碼模型來提供）。 該程式碼接著可以在雲端採用的過程中重構，以改善作業、安全性和管理 postures。

**擴充的範圍範例：** 對於採用方案呼叫中期目標（在24個月內）以裝載超過**1000 個資產（應用程式、基礎結構或資料資產）在雲端中**的小組而言，建議使用登陸區域的更健全觀點。 在這些情況下，您應該在初始登陸區域交談期間考慮管理和管理方法。 不過，這個更深入的考慮通常會在雲端採用方案中增加數周或數個月的時間。 為了將對商務結果的影響降至最低，採用小組應該平行試驗雲端中的實際工作負載，以建立更成熟的登陸區域和中央架構解決方案。

## <a name="balance-during-the-migrate-phase"></a>遷移階段期間的餘額

在遷移的過程中，採用小組通常會假設工作負載會在雲端中以目前的現有設定進行重新裝載。 這會直接與向前查看的觀點競爭，以重新架構每個工作負載，以更充分利用雲端功能。 不過，這兩個不是互斥的，而且可以在透過通用進程管理時免費使用。

**競爭優先順序：**

- **重新裝載：** 客戶通常會在其目前的狀態設定中，將所有資產複寫到雲端的隨即_轉移動作，_ 等同于遷移。 這會導致 IT 組合內的漂移。 這種方法也是在現有資料中心淘汰資產的最快方式。
- **重新架構：** 現代化每個工作負載的架構，會跨成本、效能和營運，將雲端的價值最大化。 不過，這種方法的速度很慢，而且通常需要存取每個應用程式的原始程式碼。

**最小範圍：** 在初期規劃期間，請使用重新裝載選項進行規劃，並清楚瞭解此選項是初步的商業假設，而不是技術決策。 在遷移方法中，雲端採用小組接著會針對每個遷移的工作負載挑戰這項假設。 此方法會針對每個工作負載或群組或工作負載建立遷移 factory，遵循評估/遷移/升級方法。 在評估階段，採用小組會評估每個工作負載的技術配合和架構。 該評估工作不常產生單純的隨即轉移方法，因為架構中的許多元件通常會針對重構和現代化進行選取。

**擴充的範圍範例：** 針對任務關鍵性或高敏感度的工作負載（例如大型主機或多層式微服務應用程式），在評估階段中可能需要更深入的工作負載評估。 在這些改造情況下，客戶應該使用 Microsoft Azure 良好架構的審查，並[Microsoft Azure 架構完善的架構](https://docs.microsoft.com/azure/architecture/framework)，以在評估期間調整工作負載需求。

## <a name="balance-during-the-innovate-phase"></a>創新階段期間的餘額

客戶對向的創新在規劃的功能集和客戶理解開發程式之間，建立了常見的衝突優先順序。

**競爭優先順序：**

- **功能焦點：** 創新的初始計畫會建置於現有的數位資產和雲端功能上，以提供一組符合客戶需求的功能。 您可以輕鬆地讓計畫推動技術的實行，進而著重于功能重點的開發工作。 這種方法通常會造成暫時的專案關係人滿意度，但會降低影響客戶行為的創新可能性。
- **客戶理解：** 初始計畫是開發商務端很重要的一部分，而且應該包含在一般報告中。 不過，以客戶的理解來學習、測量及建立，是一項更精確的創新成效。 專注于客戶的功能更可能導致短期與長期的客戶滿意度和業務衝擊。

**最小範圍：** 創新方法說明如何透過商業價值共識來整合策略和計畫。 本指南會介紹雲端原生工具，以加速每個專業領域的創新，並隨附最佳做法來實現。 最後，「程式改進」一節會示範建立客戶理解的方法，同時尊重雲端採用旅程的計畫和策略。 這種方法的重點在於利用盡可能少的技術來提供創新。

**擴充的範圍範例：** 有時候，創新可能相依于任務關鍵性或高敏感度的工作負載。 當「客戶」是內部使用者時，在最早的反復專案期間，開發工作可能是任務關鍵性和高敏感度。 在這些情況下，採用小組應該使用 Microsoft Azure 架構良好的審查，並 Microsoft Azure 架構完善的架構，在程式初期評估先進的架構設計。

## <a name="balance-during-the-govern-phase"></a>管理階段期間的平衡

雲端治理的實務是兩個競爭優先順序之間的持續平衡：速度和靈活性與妥善管理的環境。 雲端治理小組著重于透過一致的控制來評估企業的風險並將其降至最低，並將變更降至最低。 採用小組著重于推動商業成果，這需要新的風險，而且原本就會建立變化。

**競爭優先順序：**

- **妥善控管：** 每個設計用來將風險降至最低的控制項，都會封鎖變更或限制設計選項的某些層面。 控制對妥善管理的環境而言是不可或缺的。 不過，當控制項是以隔離的方式設計和部署時，它們可能會因為他們想要避免的風險而受到損害。
- **速度與靈活性：** 速度與靈活性是數位經濟中的基本商業需求。 這兩者都需要能夠以最少的阻礙來推動創新或採用。 當變更是在治理的隔離中進行時，它會產生新的風險，可能會以非預期的方式傷害企業。

**最小範圍：** 控管方法意味著治理和採用都不應該在隔離的情況中進行。 這個方法一開始先瞭解治理專業領域，以及關於商務風險、原則和流程的交談。 作為整個雲端採用旅程的作用中成員，治理小組可以執行一組最小的護欄，以解決雲端採用方案中的有形風險。 經過一段時間，治理小組可以重構和擴充這些護欄，以符合新的風險。 這種方法會最大化學習和創新，同時將風險降至最低。

**擴充的範圍範例：** 當商務風險很高時（特別是在採用初期），雲端治理小組可能需要加速治理實施的擴充。 您可以使用相同的指引和練習來新增這個較高層級的治理，但時間可能必須加速。 在某些情況下，在部署第一個登陸區域期間，可能會需要治理的先進狀態。

## <a name="balance-during-the-manage-phase"></a>管理階段期間的餘額

過去十年來，關於營運管理的 IT 商務模型一直持續進化。 當硬體維護從它的核心價值主張進一步發展時，作業管理上的觀點也會跟著移動。 隨著 IT 增加專注于交付商業價值，營運管理團隊也會與無作業/低營運和廣泛投資的平衡相衝突。

**競爭優先順序：**

- **廣泛的投資：** 在整個環境中，在中斷的避免、快速復原和監視方面平均投資，是作業管理的傳統方法。 這種方法可能會很昂貴，有時會複製雲端廠商所提供的支援產品。
- **無作業和低營運：** 使用雲端原生作業工具，將先前由全職員工提供的重複性和週期性工作降至最低。 減少作業管理模型中的這些操作相依性，可讓員工更具價值。 這種方法本身可能會導致欠佳作業支援。

**最小範圍：** 管理方法建議建立雲端原生、無作業基準。 確認「無營運」基準不符合所有商務需求，請與企業合作以定義承諾，並更有效地配合投資。 擴充基準以符合所有工作負載的一般需求。 然後讓平臺小組或特定工作負載小組在妥善管理的環境中維護妥善管理的解決方案。

**擴充的範圍範例：** 在大部分的環境中，很少的工作負載，其商業價值證明其營運的深度投資。 在這些情況下，IT 小組可能會想要使用 Microsoft Azure 架構良好的審查，並 Microsoft Azure 架構良好的架構，以引導更深入的作業。

## <a name="balance-during-the-organize-phase"></a>組織階段期間的餘額

這篇文章中的競爭優先順序，是以商業需求提供速度和彈性的 IT 磁片磁碟機。 這項相同的轉變會顯示在組織結構（或 v 小組結構）的變更中，以加強支援商務結果。 當 IT 領導人反映小組結構時，通常會解決兩個比較的優先順序：集中式控制項與委派控制項。

**競爭優先順序：**

- **集中式控制：** 此操作模型著重于強制執行嚴格原則所需的所有控制項。 在此模型中，它可做為創新、速度和靈活性的封鎖程式。 不過，它可以確保穩定性、合規性和安全性的程度更高。
- **委派的控制項：** 在此分散式作業模型中，假設每個 DevOps 小組或商務應用程式小組都會根據商務目標所需的解決方案，提供自己的一組控制項。 在此模型中，它會提供護欄協助您將小組保持在道路上，但盡可能將強制的技術限制數目降至最低。

**最小範圍：** 大部分的組織都會經歷一段時間的一組自然演變。 組織方法概述最常見的演變系列。 建議的指引是讓小組致力於卓越的雲端中心（CCoE）結構，以提供委派的控制方法。

**擴充的範圍範例：** 在許多情況下，會觸發集中式控制項的需求。 協力廠商合規性需求和暫時性的安全性暴露是集中控制的兩個觸發程式範例。 在這些情況下，通常會需要建立限制原則和固定的固定控制項。 不過，為了讓創新和採用繼續進行，我們鼓勵中央 IT 小組根據每個工作負載的重要性和敏感度來傳遞這些控制項。 提供控制較少的環境，但不包括範圍或風險設定檔，即使在需要控制時也允許彈性。
