# <a name="caf-enterprise-scale-security-governance-and-compliance"></a>CAF 企業規模的安全性治理和合規性

## <a name="define-encryption-and-key-management"></a>定義加密與金鑰管理

加密是在 Microsoft Azure 中確保資料隱私權、合規性及資料落地的重要步驟。 這也是許多企業最重要的安全性顧慮之一。 本節涵蓋與加密和金鑰管理相關的設計考慮和建議。

**設計考慮：**

- 適用于 Azure Key Vault 的訂用帳戶和規模限制： Key Vault 具有金鑰和秘密的交易限制。 若要在一段時間內對每個保存庫進行交易節流，請參閱 ([Azure 限制](/azure/azure-resource-manager/management/azure-subscription-service-limits)) 。

- Key Vault 可提供安全性界限，因為金鑰、秘密和憑證的存取權限是在保存庫層級。 Key Vault 存取原則指派會分別授與金鑰、秘密或憑證的許可權，但不支援細微的物件層級許可權，例如特定金鑰、秘密或憑證 ([金鑰管理](/azure/security/fundamentals/data-encryption-best-practices)) 。

- 您可以將特定應用程式和工作負載特定的密碼和共用密碼，適當地 ([控制存取權](/azure/key-vault/general/best-practices)) 來隔離。

<!-- cSpell:ignore FIPS -->

- Premium Sku 可在需要硬體安全模組保護的金鑰的情況下進行優化。 基礎硬體安全性模組 (Hsm) 符合 FIPS 140-2 層級2的規範。 適用于 FIPS 140-2 層級3合規性的受控 Azure 專用 HSM，考慮支援的案例。

- 金鑰輪替和密碼到期。

  - 使用 [Key Vault (憑證](/azure/key-vault/certificates/about-certificates) 採購和簽署) 
  - 警示/通知和自動憑證續約

- 金鑰、憑證和秘密的嚴重損壞修復需求。

  Key Vault 的服務複寫和容錯移轉功能： ([可用性和冗余](/azure/key-vault/general/disaster-recovery-guidance)) 。

- 監視金鑰、憑證和秘密使用方式。

  使用 key vault 或 Azure 監視器 Log Analytics 工作區偵測未經授權的存取： ([監視和警示](/azure/key-vault/general/alert)) 

- 委派 Key Vault 具現化和特殊許可權存取： ([安全存取](/azure/key-vault/general/secure-your-key-vault)) 

- 使用客戶管理的金鑰進行原生加密機制（例如 Azure 儲存體加密 (SSE) ）周圍的需求：
  - [客戶管理的金鑰](/azure/storage/common/storage-encryption-keys-portal)。
  - 虛擬機器 (Vm) 的完整磁片加密。
  - 傳輸中的資料加密。
  - 待用資料加密。

**設計建議：**

- 使用同盟 Azure Key Vault 模型來避免交易規模限制。

- 布建已啟用虛刪除和清除原則的 Azure Key Vault，以允許已刪除物件的保留保護。

- 藉由限制授權將金鑰、秘密和憑證永久刪除到特殊的自訂 Azure Active Directory 角色，來遵循最低許可權的模型。

- 使用公開憑證授權單位單位將憑證管理和更新程式自動化，以簡化系統管理。

- 建立金鑰和憑證輪替的自動化進程。

- 啟用保存庫上的防火牆和虛擬網路服務端點，以控制金鑰保存庫的存取權。

- 使用平臺中央 Azure 監視器 Log Analytics 工作區，以在每個 Key Vault 實例中審核金鑰、憑證和秘密的使用方式。

- 使用 Azure 原則來強制執行一致的相容設定，以委派 Key Vault 具現化和特殊許可權存取。

- 預設為主要加密功能的 Microsoft 管理金鑰，並在必要時使用客戶管理的金鑰。

- 請勿使用應用程式金鑰或秘密的集中式 Key Vault 實例。

- 請勿在應用程式之間共用 Key Vault 實例，以避免跨環境共用秘密。

## <a name="planning-for-governance"></a>規劃治理

治理提供多項機制和流程，以便維持控制 Azure 中的應用程式與資源。 Azure 原則在企業技術資產中確保安全性與合規性是不可或缺的。 它可以在 Azure 平臺服務中強制執行重要的管理和安全性慣例，以及補充以角色為基礎的存取控制，以控制授權使用者可執行檔動作。

**設計考慮：**

- 判斷需要哪些 Azure 原則

- 強制執行管理和安全性慣例，例如使用私人端點

- 若要管理和建立原則指派，可以在多個繼承的指派範圍重複使用原則定義。 您可以在管理群組、訂用帳戶和資源群組範圍中，進行集中式的基準原則指派。

- 合規性報告和審核，以確保持續符合規範

- Azure 原則有限制，例如任何指定範圍的定義限制： ([原則限制](/azure/azure-resource-manager/management/azure-subscription-service-limits)) 

- 合規性政策，例如健康保險流通與責任法案、支付卡產業 (PCI) 、資料安全性標準、服務組織控制 (SOC) 信任服務主體和準則。

**設計建議：**

- 識別所需的 Azure 標記，並使用附加原則模式來強制使用

- 將法規和合規性需求對應至 Azure 原則定義和 Azure AD 角色型存取控制指派

- 在最上層根管理群組中建立 Azure 原則定義，以便在繼承的範圍指派這些定義

- 視需要使用最底層的排除專案來管理最高層級的原則指派

- 使用 Azure 原則來控制訂用帳戶和/或管理群組層級的資源提供者註冊

- 使用內建原則，盡可能將作業的額外負荷降到最低

- 在指定範圍指派內建原則參與者角色，以啟用應用層級的治理

- 限制在根管理群組範圍中進行的 Azure 原則指派數目，以避免透過繼承範圍的排除專案進行管理

## <a name="define-security-monitoring-and-an-audit-policy"></a>定義安全性監視和稽核原則

企業必須能夠看到其技術雲端資產中發生的狀況。 Azure 平臺服務的安全性監視和審核記錄是可擴充架構的重要元件。

**設計考慮：**

- Audit data 的資料保留期限;Azure AD 報表 (premium) 有30天的保留期間

- 記錄的長期封存，例如 Azure 活動記錄、VM 記錄和平臺即服務 (PaaS) 記錄檔

- 透過 Azure in guest VM 原則進行基準安全性設定

- 重大弱點的緊急修補

- 修補離線一段長時間的 Vm

- 即時監視和警示的需求

- 安全性資訊與事件管理與 Azure 資訊安全中心和 sentinel 的整合

- Vm 的弱點評估

**設計建議：**

- 使用 Azure AD 報告功能來產生存取控制審核報告

- 若有需要，請將 Azure 活動記錄匯出至 Azure 監視器記錄以進行長期資料保留，並匯出至 Azure 儲存體以進行長期儲存，以供長期儲存到兩年以上

- 使用 Azure 原則確保合規性，為所有訂用帳戶啟用安全中心 (標準 SKU) 

- 透過 Azure 監視器記錄和 Azure 資訊安全中心監視基礎 OS 修補漂移

- 使用 Azure 原則透過 VM 擴充功能自動部署軟體設定，並強制執行符合規範的基準 VM 設定

- 透過 Azure 原則監視 VM 安全性設定漂移

- 將預設資源設定連接到集中式 Azure 監視器 Log Analytics 工作區

- 使用事件方格型解決方案來進行記錄源即時警示

## <a name="plan-for-platform-security"></a>規劃平臺安全性

在您採用 Azure 時，維持狀況良好的安全性狀態是不可或缺的。 除了可見度之外，您還必須能夠控制 Azure 服務發展時的初始設定和變更。 因此，規劃平臺安全性是關鍵的。

**設計考慮：**

- 共同責任

- 高可用性和災害復原

- 在資料管理和控制平面作業方面，以一致的方式在 Azure 服務之間進行安全性

- 主要平臺元件的多租使用者，從 Hyper-v 和 Hsm 底層 Key Vault 到資料庫引擎

**設計建議：**

- 您應在基礎需求的內容中，進行每個必要服務的聯合檢查。 如果您想要攜帶您自己的金鑰，則可能不會在所有已被視為的服務上受到支援。 您必須實行相關的緩和措施，讓不一致的結果不會阻礙預期的結果。 選擇適當的區域配對和嚴重損壞修復區域，以將延遲降至最低。

- 開發安全性允許清單計畫來評估服務安全性設定、監視、警示，以及如何將這些服務與現有的系統整合。

- 在允許之前，請先判斷 Azure 服務的事件回應計畫。

- 使用 Azure AD 報告功能來產生存取控制審核報告。

- 將您的安全性需求與 Azure 平臺藍圖保持一致，以隨時掌握最新發行的安全性控制措施。

- 在適當的情況下，實以零信任方式存取 Azure 平臺。
