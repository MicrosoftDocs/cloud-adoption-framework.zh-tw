# <a name="standard-enterprise-governance-guide-improve-the-security-baseline-discipline"></a>標準企業治理指南：改善安全性基準專業領域

本文藉由新增可支援將受保護資料移至雲端的安全性控制，來推進 [治理策略的敘述](narrative.md) 。

## <a name="advancing-the-narrative"></a>前進敘述

IT 和業務領導力很滿意 IT、應用程式開發和 BI 團隊的早期實驗結果。 若要實現這些實驗中的具體商業價值，必須允許這些小組將受保護的資料整合到解決方案中。 此整合會觸發對公司原則的變更。 它也需要在受保護的資料進入雲端之前，對雲端治理實施進行累加式改進。

### <a name="changes-to-the-cloud-governance-team"></a>雲端治理小組的變更

由於目前為止所提供變更敘述和支援的影響，雲端治理小組現在以不同的方式來查看。 發起小組的兩位系統管理員現在被視為經驗豐富的雲端架構設計師。 當此敘述發展時，其認知將會從雲端監管者轉移到更多雲端守護者角色。

差異很微妙，但在您建立以治理為焦點的 IT 文化時，這是一項重要的區別。 雲端保管者會清除創新雲端架構設計人員所做的混亂。 這兩個角色具有自然的分歧和相反的目標。 另一方面，雲端守護者可協助保持雲端安全，讓其他雲端架構設計人員可以更快速地移動，但混亂更少。 雲端守護者與建立可加速部署和採用的範本有關。 因此除了防禦者雲端治理的五個專業領域之外，它們還是創新的加速器。

### <a name="changes-in-the-current-state"></a>處於目前狀態的變更

在此敘述開始時，應用程式開發小組仍然致力於開發/測試能力，而 BI 小組仍處於實驗階段。 它會操作兩個託管的基礎結構環境，稱為生產和 DR。

從那時起，某些將會影響治理的事項已經改變：

- 應用程式開發小組已實行 CI/CD 管線，以改良的使用者體驗來部署雲端原生應用程式。 該應用程式尚未與受保護的資料互動，因此它不是生產環境就緒。
- IT 內的商業智慧小組會從物流、庫存和協力廠商來源主動策劃了雲端中的資料。 這項資料是用來推動可塑造商務流程的新預測。 在將客戶和財務資料整合到資料平臺之前，這些預測和見解都無法採取行動。
- IT 小組正在進行 CIO 和 CFO 規劃，以淘汰 DR 資料中心。 DR 資料中心內的 2,000 個資產中，有 1,000 多個資產已經遭到淘汰或移轉。
- 已現代化個人資料和財務資料的鬆散定義原則。 新的公司原則會因應相關的安全性和治理原則的實施。 小組仍然停滯不前。

### <a name="incrementally-improve-the-future-state"></a>以累加方式改進未來的狀態

應用程式開發和 BI 小組的早期實驗顯示客戶體驗和資料驅動決策的潛在改進。 兩個小組都希望將這些解決方案部署到生產環境，藉此在接下來的 18 個月內擴展雲端的採用率。

在剩餘的六個月期間，雲端治理小組將會實施安全性和治理需求，以允許雲端採用小組在這些資料中心遷移受保護的資料。

目前和未來狀態的變更會產生新風險，需要新的原則聲明。

## <a name="changes-in-tangible-risks"></a>有形風險的變更

**資料缺口：** 當您採用任何新的資料平臺時，與潛在資料缺口相關的責任會有固有的增加。 採用雲端技術的技術人員會有更多的責任，以實行可降低此風險的解決方案。 為確保這些技術人員會履行那些責任，必須實作強固的安全性和治理策略。

此業務風險可能會延伸出少數技術風險：

- 任務關鍵性應用程式或受保護的資料可能會不慎部署。
- 由於加密決策不佳，因此可能會在儲存體期間公開受保護的資料。
- 未經授權的使用者可能會存取受保護的資料。
- 外部入侵可能會導致受保護的資料遭到存取。
- 外部入侵或拒絕服務攻擊可能會導致業務中斷。
- 組織或雇用變更可能會允許未經授權存取受保護的資料。
- 新的惡意攻擊可能會創造入侵或存取的機會。
- 不一致的部署程序可能導致安全性漏洞，進而造成資料外洩或中斷。
- 設定漂移或遺失修補程式可能會導致非預期的安全性漏洞，進而造成資料外洩或中斷。

**資料遺失：** 在新的平臺中，資料遺失也有潛在的風險。 安全性和治理策略應考慮可能發生資料遺失的下列情況：

- 任務關鍵性資源遺失或刪除。
- 有任務關鍵性資源存在，但資料因為意外刪除而遺失。
- 有任務關鍵性資源存在，但由於惡意系統管理，資料會遺失。

## <a name="incremental-improvement-of-policy-statements"></a>原則語句的累加式改進

下列原則變更將有助於補救新的風險和指南的實施。 清單看起來很長，但採用這些原則可能比您想像的還簡單。

- 所有已部署的資產都必須依據嚴重性和資料分類來分類。 雲端治理小組和應用程式擁有者必須先審查這些分類，然後再部署至雲端。
- 儲存或存取受保護資料的應用程式，其管理方式必須與不是的應用程式不同。 至少應該劃分這些資料，以避免非預期地存取受保護的資料。
- 所有受保護的資料在待用時都必須經過加密。 此加密是所有 Azure 儲存體帳戶的預設值。 但是，當您在 VM 中使用 SQL (TDE 和資料行加密) 時，您可能需要額外的加密策略，包括加密儲存體帳戶內的資料、Vm 的加密，以及資料庫層級加密。
- 您可以不小心刪除任務關鍵性資料。 您必須開發資料備份策略來處理此風險，並在刪除點之前還原資料。 惡意系統管理員也可以刪除任務關鍵性資料及其備份。 若要處理此案例，備份資料的刪除應該是可以反轉的虛刪除。 Azure 備份可協助這兩種情況。
- 任何包含受保護資料之區段中的較高許可權都應該是例外狀況。 任何這類例外狀況都會與雲端治理小組一起記錄並定期進行審核。
- 包含受保護資料的網路子網必須與其他子網隔離。 受保護資料子網路之間的網路流量將定期稽核。
- 任何包含受保護資料的子網都應可透過公用網際網路或跨資料中心直接存取。 這些子網路的存取必須透過中繼子網路進行路由。 對那些子網路的所有存取都必須經過防火牆解決方案，該解決方案可以執行封包掃描和封鎖功能。
- 治理工具必須稽核並強制執行安全性管理小組所定義的網路設定需求。
- 治理工具必須限制只對已核准的映射進行 VM 部署。
- 治理程序必須確認任務關鍵性應用程式和受保護資料的備份、復原和 SLA 遵循皆正確實作。
- 若有可能，節點組態管理應該將原則需求套用到任何客體作業系統的設定。
- 治理工具必須強制在所有已部署的資產上啟用自動更新。 必須與作業管理小組一起檢閱違規事件，並根據作業原則來進行修復。 未自動更新的資產必須包含在 IT 作業所擁有的處理常式中。
- 為任何任務關鍵性應用程式或受保護的資料建立新的訂用帳戶或管理群組，將需要雲端治理小組的評論，以確保已指派適當的藍圖。
- 最小許可權的存取模型將會套用到包含任務關鍵性應用程式或受保護資料的任何管理群組或訂用帳戶。
- 安全性小組應定期檢閱可能影響雲端部署的趨勢與攻擊，以更新雲端中使用的安全性管理工具。
- 部署工具必須由雲端治理小組核准，以確保持續治理已部署的資產。
- 部署腳本必須在雲端治理小組可存取的中央存放庫中進行維護，以進行定期審核和審核。
- 治理流程必須包含部署期間和一般週期的稽核，以確保所有資產之間的一致性。
- 部署任何需要客戶驗證的應用程式，都必須使用與內部使用者的主要身分識別提供者相容的核准身分識別提供者。
- 雲端治理程序必須包含身分識別管理小組的每季審查。 這些審查有助於識別惡意執行者或雲端資產設定應該避免的使用模式。

## <a name="incremental-improvement-of-governance-practices"></a>治理做法的累加式改進

治理 MVP 設計將會變更，以包含新的 Azure 原則以及 Azure 成本管理 + 計費的實施。 這兩個設計變更將共同實現新的公司原則聲明。

- 網路和 IT 安全性小組將會定義網路需求。 雲端治理小組將會支援交談。
- 身分識別和 IT 安全性小組將會定義身分識別需求，並對本機 Active Directory 的執行進行任何必要的變更。 雲端治理小組將會審查變更。
- 在 Azure DevOps 中建立存放庫來存放所有相關的 Azure Resource Manager 範本和指令碼式的組態，並為這些項目設定版本。
- Azure 復原服務保存庫的執行：
    - 定義及部署用於備份和復原程式的 Azure 復原服務保存庫。
    - 建立 Resource Manager 範本以在每個訂用帳戶中建立保存庫。
- Azure 資訊安全中心實作：
    - 為包含受保護資料分類的任何管理群組設定 Azure 資訊安全中心。
    - 依預設，將自動佈建設定為開啟，以確保修補的合規性。
    - 建立 OS 安全性設定。 IT 安全性小組將會定義設定。
    - 在首次使用安全中心時支援 IT 安全性小組。 將安全性中心的使用轉換給 IT 安全性小組，但維持存取權，以持續改善治理。
    - 建立 Resource Manager 範本，此範本會反映訂用帳戶內資訊安全中心設定所需的變更。
- 更新所有訂用帳戶的 Azure 原則：
    - 在所有管理群組和訂用帳戶中，對資料和資料分類的重要性進行審核並強制執行，以識別具有受保護資料分類的任何訂閱。
    - 僅稽核並強制使用已核准的映像。
- 針對包含受保護資料分類的所有訂用帳戶更新 Azure 原則：
    - 僅稽核並強制使用標準 Azure RBAC 角色。
    - 為個別節點上待用的所有儲存體帳戶和檔案，稽核並強制執行加密。
    - 稽核並強制執行使用 NSG 連到所有 NIC 和子網路的應用程式。 網路和 IT 安全性小組將會定義 NSG。
    - 針對每個網路介面，審核並強制使用已核准的網路子網和虛擬網路。
    - 稽核並強制執行使用者定義的路由表限制。
    - 套用適用于來賓設定的內建原則，如下所示：
        - 稽核 Windows 網頁伺服器目前使用安全的通訊協定。
        - 稽核 Linux 及 Windows 電腦內密碼安全性設定皆設定正確。
    - 審核並強制執行訂用帳戶中的 Azure 復原服務保存庫。
- 防火牆組態：
    - 識別符合所需安全性需求的 Azure 防火牆設定。 或者，識別與 Azure 相容的相容協力廠商應用裝置。
    - 建立 Resource Manager 範本來部署具有必要設定的防火牆。
- Azure 藍圖：
    - 建立名為 protected data 的新藍圖。
    - 將防火牆、Azure 資訊安全中心範本和 Azure 復原服務保存庫範本新增至藍圖。
    - 為受保護資料的訂用帳戶新增原則。
    - 將藍圖發佈到目前計畫裝載受保護資料的任何管理群組。
    - 將新的藍圖套用至每個受影響的訂用帳戶和現有的藍圖。

## <a name="conclusion"></a>結論

將上述的程式和變更新增至治理 MVP，將有助於修復與安全性治理相關聯的許多風險。 同時，它們會新增網路、身分識別，以及保護資料所需的安全性監視工具。
