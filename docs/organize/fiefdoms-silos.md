# <a name="organizational-antipatterns-silos-and-fiefdoms"></a>組織反模式：定址接收器和領域

對於商業實務、文化特性或技術作業的任何重大變更，都需要有成長思維。 成長思維的核心是接受變更，並讓您能夠在不明確的情況中產生。

有些反模式可能會在想要成長和轉換的組織中封鎖成長思維，包括 micromanagement、偏差思維和排他性行為實務。 其中許多阻礙都是個人的挑戰，為每個人創造個人成長的機會。 但是，其中兩個常見的反模式需要的不只是個別成長或成熟度：定址接收器和領域。

![狀況良好的團隊與組織反模式的比較](../_images/ready/fiefdom-and-silo.png)

這些反模式是在各種小組內進行變動的結果，這會導致組織行為狀況不良。 若要解決每個反模式所造成的阻力，請務必瞭解這種形式的根本原因。

## <a name="healthy-organic-it-teams"></a>狀況良好、有機的 IT 小組

在 IT 之間建立一項人力是很自然的。 建立具有類似專業知識、共用流程、共同目標和一致願景的團隊是狀況良好的。 這些小組也可以擁有自己的 microculture、共用的規範和觀點。

狀況良好的 IT 團隊將焦點放在與其他團隊合作，以提升其職責的成功完成。 狀況良好的 IT 團隊想要瞭解其技術貢獻設計來支援的商務目標。 詳細資料和會計影響可能是模糊的，但小組的價值貢獻通常是在小組內部理解。

雖然狀況良好的 IT 團隊對他們所支援的技術有熱情，但他們已開放變更且願意試用新事物。 這類團隊通常是最早且最強的 [雲端中心 (CCoE) ](./cloud-center-of-excellence.md) 工作。 他們的貢獻應該很強烈地建議。

### <a name="natural-resistance-to-change"></a>要變更的自然阻力

有時候，狀況良好的 IT 團隊 microcultures，可能會對主管或由上而下的決策做出不良的反應來推動變更。 這種反應很自然，因為集合項 nics 與共享的規範通常會互相合作來克服外部威脅。

影響小組日常工作、安全性概念或自主性的變更，都可以視為集體的風險。 阻力的徵兆通常是早期的指標，因為小組成員不會覺得這些成員是決策制定程式的一部分。

當雲端架構設計人員和其他領導人投資 abolishing 個人偏差和駕駛以納入現有的 IT 小組時，這項阻力可能會在一段時間內快速減少並進行溶解。 雲端架構設計人員和領導人可使用的一項工具來建立包容性決策，是 CCoE 的構成。

### <a name="healthy-friction"></a>狀況良好的摩擦

您可以輕鬆地將阻力與摩擦混淆。 現有的 IT 團隊對於過去的錯誤、具體風險、部落解決方案的知識，以及未記載的技術債務，都有很豐富的知識。 可惜的是，即使是 healthiest IT 團隊，也可以將這些重要資料點描述為特定技術解決方案的一部分，而不應該變更。 這種通訊方式會遮罩團隊的知識，並建立阻礙的認知。

提供這些小組在未來的術語中進行通訊的機制，將會增加資料點、找出間距，並在提議的解決方案周圍建立狀況良好的分歧。 這項額外的衝突將會將解決方案的粗略邊緣向下，並驅動長期的價值。 只要變更對話，就能更清楚地解決複雜的主題，並產生能源來提供更成功的解決方案。

[定義公司原則](../govern/corporate-policy.md)的指導方針旨在促進與商務專案關係人的風險型交談。 但是，這個相同的模型可以用來協助與可感知雲端的團隊進行交談。 當阻力的認知很廣泛時，在 [雲端治理小組](./cloud-governance.md)的職責中包含電阻解析實務可能是明智的做法。

## <a name="antipatterns"></a>反模式

IT 部門創造出狀況良好的 IT 團隊，也可能導致反模式封鎖轉換和雲端採用。 IT 定址接收器和領域與健康 IT 團隊內的自然 microcultures 不同。 在任一種模式中，小組焦點通常會導向保護其「turf」。 當小組成員面對推動變更和改進營運的機會時，他們將花費更多的時間和精力來封鎖變更，而不是尋找正面的解決方案。

如先前所述，狀況良好的 IT 小組可以建立自然抵抗和正面摩擦。 定址接收器和領域是不同的挑戰。 這兩個反模式都沒有記載的前置指標。 這些反模式通常會在 [雲端中心的](./cloud-center-of-excellence.md) 各月和 [雲端治理小組](./cloud-governance.md) 工作之後識別。 系統會將它們視為持續抵抗的結果。

即使是在有毒文化特性中，CCoE 和雲端治理小組的工作也應該協助推動文化成長和技術進展。 經過幾個月之後，幾個小組可能仍不會顯示任何內含行為的徵兆，也可能不會顯示其抵抗的變化。 這些小組可能會在下列其中一種反模式模型中運作：定址接收器和領域。 雖然這些模型的徵兆很類似，但根本原因和解決阻力的方法在兩者之間有截然不同的差異。

## <a name="it-silos"></a>IT 定址接收器

IT 接收器的團隊成員很可能會透過其與少數 IT 廠商或技術特製化領域的一致來自行定義。 但請勿將接收器與 IT 領域混淆。 定址接收器通常是透過緩和和熱情來推動，而定址接收器通常比領域背後的恐懼驅動動機更容易克服。

這種反模式通常是由特定解決方案的一般熱情所形成。 然後，小組的 advanced 技能會加強 IT 定址接收器，因為這是該特定解決方案的投資結果。 這種優質的技能可作為雲端採用工作的加速器，如果可以克服變更的阻力。 如果接收器已細分或小組成員無法精確地評估選項，它也可以成為主要封鎖者。 幸運的是，IT 定址接收器通常可以在不需要對組織結構進行任何重大變更的情況下克服。

### <a name="address-resistance-from-it-silos"></a>解決 IT 接收器的阻力

您可以透過下列方法來處理 IT 定址接收器。 最佳方法將取決於阻力的根本原因。

**建立虛擬小組：** 雲端採用架構的「 [組織準備就緒](./index.md) 」一節描述了多層式結構，可用於整合和定義四個虛擬團隊。 此結構的其中一個優點是跨組織可見度和包含。 「 [卓越的雲端中心](./cloud-center-of-excellence.md) 」會建立高階工程師想要參與的理想團隊。 這有助於建立新的跨解決方案對齊，而不受組織結構的限制，並且會推動 IT 接收器 sheltered 的頂尖工程師。

推出 [雲端策略小組](./cloud-strategy.md) 將會立即看到有關雲端採用工作的 IT 貢獻。 當它將對抗分開時，此可見度有助於鼓勵 IT 和企業領導人適當地支援這些不當的團隊成員。 此程式是專案關係人參與和支援的快速途徑。

**考慮實驗和暴露：** IT 定址接收器中的小組成員可能會被限制為一段時間的特定方式。 若要解決阻礙的第一步，請特別注意一項重點。

測試和暴露是功能強大的工具，可在接收器中細分阻礙。 小組成員可能會抵抗競爭解決方案，因此不會將其視為與其現有解決方案競爭的實驗。 但在雲端的第一個工作負載測試中，組織應該實行競爭解決方案。 孤立的團隊應受邀參與做為輸入和審核來源，但不能作為決策人員。 這應該會清楚地傳達給小組，以及在移至生產環境解決方案之前，讓小組更深入地參與團隊的承諾。

在探討競爭解決方案的過程中，請使用「 [定義公司原則](../govern/corporate-policy.md) 」中所述的實務來記載實驗的具體風險，並建立原則，協助孤立的團隊更熟悉未來的狀態。 這會將小組公開至新的解決方案，並強化未來的解決方案。

「**不界限」：** 推動雲端採用的團隊可以探索絕佳的新雲端原生解決方案，輕鬆地推送界限。 這是移除界限的一半方法。 但是，這種考慮可以進一步強化 IT 的定址接收器。 快速推送以進行變更，而不考慮現有的文化特性，可能會產生狀況不良的衝突，並導致自然阻力。

當它的定址接收器開始抵禦時，請務必在您自己的解決方案中「界限較少」。 請注意一個簡單的事實：雲端原生不一定是最佳的解決方案。 請考慮混合式解決方案，以提供機會將 IT 接收器的現有投資延伸到未來。

也請考慮 IT 接收器小組現在使用的雲端式解決方案版本。 使用這些解決方案進行實驗，並將您自己暴露在 IT 接收器中的觀點。 您至少會得到新的觀點。 在許多情況下，您可能會獲得足夠的 IT 接收器以減輕阻力的考慮。

**投資教育：** IT 接收器的許多人都熱衷於目前的解決方案，因為擴充自己的教育的結果。 投資這些團隊的課程很少放在一起。 配置這些人的時間來參與自學、類別或甚至是會議，以將日常焦點放在目前的解決方案上。

為了讓教育成為投資，某些報酬率必須是費用的結果。 在用於投資的 exchange 中，小組可能會向與雲端採用相關的其餘小組示範提議的解決方案。 它們也可以提供具體風險、風險管理方法，以及採用建議解決方案的預期原則檔。 每個小組都會在解決方案中參與這些小組，並協助您充分利用他們的部落知識。

將 **障礙轉為速度：** IT 定址接收器可能會變慢或停止任何轉換。 測試和反復專案將會發現一種方法，但只有在專案持續移動的情況下才會進行。 專注于將障礙變成隻是加速的。 定義每個人都可以在 exchange 中暫時熟悉的原則，以持續進展。

比方說，如果安全性是障礙，因為它的安全性解決方案無法監視雲端中受保護資料的危害，請建立資料分類原則。 在找不到比較容易解決方案之前，避免將已分類的資料部署至雲端。 使用混合式或雲端原生解決方案來監視受保護的資料，以邀請 IT 安全性進入實驗。

如果網路小組以定址接收器的形式運作，請找出本身獨立且沒有網路相依性的工作負載。 在使用混合式或替代解決方案時，以平行方式進行實驗、公開和教育網路團隊。

**請耐心等候並包含：** 在不支援 IT 接收器的情況下繼續進行，是很吸引人的。 但是，這種決策會導致中斷和阻礙。 變更 IT 接收器成員的思維可能需要一些時間。 請耐心等候他們的自然阻力--將它轉換為價值。 包含並邀請狀況良好的摩擦，以改善未來的解決方案。

**永不競爭：** IT 定址接收器有原因。 因為原因，所以會持續保存。 維護小組成員所熱衷於的解決方案有一項投資。 直接與解決方案或 IT 接收器競爭，將會干擾實現業務成果的真正目標。 這個陷阱已封鎖許多轉換專案。

保持專注于目標，而不是目標的單一元件。 協助強調 IT 接收器解決方案的正面部分，並協助小組成員對未來的最佳解決方案做出明智的決策。 請勿 insult 或降低目前的解決方案，因為這會是這些。

**與企業合作：** 如果 IT 定址接收器未封鎖業務成果，您為什麼要在意？ 沒有完美的解決方案或完美的 IT 廠商。 有競爭原因，每個都有自己的優點。

藉由支援和與強大的 [雲端策略小組](./cloud-strategy.md)合作，來採用多樣性並納入企業。 當 IT 定址接收器支援封鎖業務成果的解決方案時，在沒有技術 squabbles 雜訊的情況下，可以更輕鬆地傳達該障礙。 支援非封鎖 IT 定址接收器將會為所要的業務成果，提供合作夥伴的能力。 當 IT 接收器呈現合法的封鎖程式時，這些工作會從企業獲得更多尊重和更高的支援。

## <a name="it-fiefdoms"></a>IT 領域

IT fiefdom 中的小組成員可能會透過其與特定流程或職責區域的一致來自行定義。 小組會假設外部對其責任領域的影響將會導致問題。 領域通常是恐懼導向的反模式，需要有巨大的領導能力支援來克服。

領域在縮減、經常 turbulence IT 人員或 IT 領導力不良的組織中特別常見。 當企業將它純粹視為成本中心時，領域會更有可能發生。

一般情況下，領域是一種會導致小組遺失以及相關聯的耗電量的行經理結果。 這些領導人通常對其小組有意義，而且需要保護其附屬專案免于負面的後果。 像「收容所團隊的變更」和「從程式中斷保護小組」這類片語可能是過度受監管管理員的指標，可能需要領導階層的更多支援。

### <a name="address-resistance-from-it-fiefdoms"></a>解決 IT 領域的阻力

IT 領域可以遵循 [解決 it 接收器阻力](#address-resistance-from-it-silos)的方法，來示範一些成長。 在您嘗試解決來自 IT fiefdom 的阻力之前，建議您先將小組視為 IT 接收器。 如果這些類型的方法無法產生任何重大變更，則抵抗團隊可能會遭受 IT fiefdom 反模式的影響。 IT 領域的根本原因有點複雜，因為這種阻力通常是來自直接線路管理員 (或領導人較高的) 。 IT 定址接收器導向的挑戰通常更容易克服。

當 IT 的阻礙領域封鎖雲端採用工作時，可能會結合投入時間來評估現有 IT 領導人的情況。 IT 領導人應該仔細考慮 [雲端策略小組](./cloud-strategy.md)、 [雲端中心](./cloud-center-of-excellence.md)和 [雲端治理小組](./cloud-governance.md) 的見解，再進行決策。

> [!NOTE]
> IT 領導人應該不會稍微變更組織結構。 他們也應該驗證並分析每個支援小組的意見反應。 但是雲端採用的轉換工作，通常會在這種情況下，放大已不容易察覺或尚待發揮的根本問題。 當領域導致公司成功時，領導力的變更可能是必要的。
>
> 幸運的是，移除 fiefdom 的領導者通常不會結束終止。 這些強大的熱衷於領導人通常會在一段短暫的反映後進入管理角色。 有了正確的支援，這項變更可能會對 fiefdom 和目前團隊的領導人有良好的狀況。

<!-- -->

> [!CAUTION]
> 對於 IT 領域的管理員而言，保護小組免于風險是明確的領導價值。 但保護和隔離之間有一個細微的線。 當小組遭到封鎖而無法參與駕駛變革時，可能會對小組造成心理和專業的後果。 強烈的變更可能會很強，尤其是在可見變更的時候。
>
> 任何隔離小組的經理都可以透過實驗上述各節中與狀況良好的 IT 小組相關聯的指導方針，以最佳方式示範成長思維。 主動和開放式參與治理和 CCoE 活動可能會導致個人成長。 IT 領域的主管最適合用來變更 stifling 心態，並協助小組開發新的想法。

IT 領域可能是有問題的領導力問題的跡象。 為了克服 IT fiefdom，IT 領導人需要能夠變更營運、職責，甚至是在提供特定小組的程式碼的人員。 當這些變更都需要時，最好使用清楚且可防禦的資料點來處理這些變更。

可能需要與商務專案關係人、業務動機和業務成果保持一致，才能推動必要的變更。 與 [雲端策略小組](./cloud-strategy.md)、 [雲端卓越中心](./cloud-center-of-excellence.md)和 [雲端治理小組](./cloud-governance.md) 合作，可以提供可防禦位置所需的資料點。 必要時，這些小組應參與群組擴大，以解決因 IT 領導力而無法解決的問題。
