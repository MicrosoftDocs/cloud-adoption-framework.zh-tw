# <a name="get-started-build-a-cloud-strategy-team"></a>開始使用：建立雲端策略小組

每個雲端採用旅程都需要某種程度的策略才能成功。 本入門指南可協助您建立專用的小組或虛擬小組（v 小組），以提供穩固的雲端策略。 第一個步驟是決定您是否需要策略小組，或現有小組成員是否能夠以分散式責任提供雲端策略。

建立雲端策略小組來定義動機和業務結果，並驗證和維護商務優先順序與雲端採用工作之間的一致。 當商務成果影響商務功能時，策略小組應該包含整個組織的商務領導人。 雲端策略小組的目標是要產生雲端技術所啟用的有形商業成果。 整體來說，這個小組可確保雲端採用工作的進度與商務成果一致。 可能的話，商務結果和雲端策略小組都應該在程式中及早定義。

![開始建立雲端策略小組](../../_images/get-started/strategy-team-map.png)

> [!NOTE]
> 本文將提及策略主持者。 當雲端策略小組開始使用時，策略主持者會協助團隊保持一致，並暫時保有建立對齊的責任。 策略主持者通常是最負責雲端採用旅程之成功的人員。 此角色也經常由程式經理、架構設計師或顧問負責。

## <a name="step-1-determine-if-a-cloud-strategy-team-is-needed"></a>步驟1：判斷是否需要雲端策略小組

雲端策略小組在雲端中提供必要的功能，稱為「雲端策略」功能。 形成雲端策略小組需要一組已定義的專屬商業領導人、專案關係人和計畫經理，以週期性週期性步調來推動雲端採用的策略。

**項**

- 判斷您是否需要雲端策略小組。

**支援交付後完成的指引：**

如果雲端採用是匯入企業、需要進行中的主管支援，或面臨企業的風險，則此小組通常是一項需求。

|                                                     |                                                                                                                            |

| **對企業很重要：** | <li> 雲端採用工作具有面板層級的可見度。 <li> 雲端採用工作的成功，將可改善市場定位、客戶保留期或收益。 <li> 採用組合中的程式會直接對應到策略性商務結果。 <li> 這項採用工作的工作負載組合是策略性、要徑任務，而且可能會影響多個業務單位。 |
| **需要進行中的主管支援：** | <li> 雲端採用工作將會影響我們管理組織變更的方式。 <li> 雲端採用工作需要多個商務使用者進行額外的訓練，而且可能會中斷商務功能。 <li> 現有的 IT 營運小組或廠商隨時都在現有的資料中心內。 <li> 現有的 IT 小組並未完全購買。 |

如果下列各項代表您現有的需求，則下列步驟將有助於建立雲端策略小組。

- 此策略主持者負責判斷是否需要雲端策略小組。

## <a name="what-if-i-dont-need-a-cloud-strategy-team"></a>如果我不需要雲端策略小組，該怎麼做？

回顧雲端策略[功能](../../organize/cloud-strategy.md)，這是提供雲端策略需求所需的。 並非每個組織都需要專屬的小組或 v 小組，才能符合其策略性需求。 在您的[RACI 範本](https://archcenter.blob.core.windows.net/cdn/fusion/management/raci-template.xlsx)中，列出策略的核心標準責任，並識別小組中將負責每個人的人員。 如果其中一個人要使用所有這些標準責任，請在 RACI 範本中，將「雲端策略」取代為該人員的名稱。

## <a name="step-2-establish-the-cloud-strategy-team"></a>步驟2：建立雲端策略小組

雲端策略小組將作為商務領導人和 IT 領導人之間的週期性對齊點。 根據重要性、風險及主管支援的層級，推動策略小組的需求，小組的參與和撰寫可能會有所不同。

**項**

- 找出適當的組織或個人，他願意共同負責推動雲端採用策略的責任和責任。

**支援交付後完成的指引：**

- 記載並分享您對上述步驟1的推理，以識別將受益于定期參與的專案關係人，並且能夠協助推動策略。
- 請參閱[雲端策略的功能](../../organize/cloud-strategy.md)，瞭解哪些人可能適合的想法。
- 回顧這項功能的[最小範圍](../../organize/cloud-strategy.md#minimum-scope)和[交付成果](../../organize/cloud-strategy.md#deliverable)，以驗證每個潛在參與者的對齊和頻寬。
- 查看各種[RACI 設定範例](../../organize/raci-alignment.md#strategic-alignment)，以根據您目前的小組結構來建立正確的 RACI 圖表。 或使用 RACI 範本隨附的其中一個範例索引標籤。
- 將結果記錄在 [組織對齊] 索引標籤下的[RACI 範本](https://archcenter.blob.core.windows.net/cdn/fusion/management/raci-template.xlsx)中。

**責任小組：**

- 「主持者」策略負責建立雲端策略小組。

## <a name="step-3-establish-a-cadence"></a>步驟3：建立步調

在雲端採用旅程的初期，此小組需要經常的互動和反復的策略審查。 當採用開始時，該頻率將會減少，並轉換成專注于待處理專案優先順序的狀態和驗證/調整。

下面的步驟4、5和6應于4到6周內完成。 其餘的步驟將在後續會議中完成。 在小組開始執行下列步驟7之前，應維持更高的頻率會議。

**項**

請參閱建議的會議步調，並安排小組參與者所有策略的會議。

**支援交付後完成的指引：**

- 請參閱建議的短期和長期[會議步調](../../organize/cloud-strategy.md#deliverable)，以配合每個已記載的參與者。

**責任小組：**

- 策略主持者負責建立雲端策略小組的步調。

## <a name="step-4-establish-a-motivation-driven-strategy"></a>步驟4：建立動機導向的策略

雲端採用旅程同時包含了遷移和創新的方法。 當技術小組定義策略時，策略通常是由小組目前的技能和優勢來推動。 這種策略可能是技術上的成功，但卻有產生限制業務影響的風險。

雲端策略小組的第一個目標是要根據業務動機定義高階策略。 這通常會在一小時的研討會中完成，其中包含所有雲端策略小組成員。 它也需要至少一小時的時間，才能使用各種技術團隊和受影響的專案關係人來審查動機。

在第一場研討會期間，小組的每個成員都應該優先考慮「[瞭解動機](../../strategy/motivations.md)」文章中的動機，並分享其最高優先順序。 策略主持者將有助於引導一或多個交談，直到主題以遷移或創新的方向顯露為止。 這兩個類別的前3個清單中可能會有一個動機，這可能需要小組在明確模式仰賴到一個方向之前，更深入瞭解其清單。

**項**

此練習將會呈現可協助建立小組成員之間對齊的交談。 此交付項將有助於引導其餘的策略和遵循計畫。

- 記錄[策略和計畫範本](https://archcenter.blob.core.windows.net/cdn/fusion/readiness/Microsoft-Cloud-Adoption-Framework-Strategy-and-Plan-Template.docx)中的動機。

**支援交付後完成的指引：**

- [瞭解動機](../../strategy/motivations.md)：重要的商務事件和一些遷移動機通常會對成本產生影響，進而提高所有後續工作的成本控制重要性。 其他與創新或透過遷移的成長相關的正向動機，可能會更專注于頂尖的收益。 瞭解動機有助於瞭解優先順序成本管理有多高。

## <a name="step-5-establish-business-outcomes"></a>步驟5：建立商務結果

雲端策略小組的每個成員都會要求定義一或多個商務結果，並具有特定計量來測量商務成功。 如果可以在雲端採用工作的直接結果中改善該計量，則會要求小組成員共用預期的影響。 如果雲端採用工作不會影響計量，但可讓企業更有效地驅動該度量，這也應該記載。

雲端策略小組的許多領導者可能需要分解其核心計量，以識別可能受到雲端採用工作影響或影響的結果。 如果小組成員的成果不會受到此旅程的影響或影響，那麼就很難在程式中維護其感興趣的層級。 主持人應該與每個領導者合作來開發一致的計量，並重新評估該小組成員是否為參與策略小組的適當人員。

對商務結果的影響可能需要一些時間。 這些變更類型的移動速度通常會比技術變更慢。 為了維護透明度，策略小組應該同意較短的文字收集計量。 這些計量可能包括可在每個小組會議審查的技術和其他變更，以示範技術目標和商務成果的進度。

**項**

- 針對雲端策略小組的每個成員至少識別一個預期的結果。
- 縮小成員的清單，以配合預期結果的預期時間承諾。
- 對齊一組短期和中間詞彙，以支援持續的進度報表。

**支援交付後完成的指引：**

- 記錄[策略和計畫範本](https://archcenter.blob.core.windows.net/cdn/fusion/readiness/Microsoft-Cloud-Adoption-Framework-Strategy-and-Plan-Template.docx)中的商務結果。
- [商務結果](../../strategy/business-outcomes/index.md)：某些會計結果通常會非常耗費成本。 當想要的結果對應至會計計量時，最好是儘早投資治理的成本管理專業領域。
- [學習度量](../../strategy/learning-metrics.md)：學習計量有助於填補商業結果與技術採用工作之間的橋樑。

**負責的小組：**

雲端策略小組負責定義具有特定計量的商務結果，這些度量可用來測量商務結果的成功。

## <a name="step-6-make-a-decision-to-proceed-or-cancel-based-on-the-business-justification"></a>步驟6：根據商業理由做出決策以繼續或取消

商業理由可以協助規劃、長期報酬和預期擁有權總成本（TCO）。 在此步驟中，雲端策略小組應同意所需的最少量分析，以協助策略小組符合向前決策。 策略性的對齊可能需要深度規劃和 TCO 分析。 大部分的雲端策略小組都會發現一個簡單的成本分析，做為適當的分析，以配合方向。

策略小組的每個成員都應該檢查商業理由的常見誤解和方法，以協助溝通支援小組所預期的特定分析。 一旦傳達該預期，策略小組就可以降低其時間投資和會議步調。 不過，小組會維護完成策略的責任，直到商業理由和數位資產分析都已達成同意為止。

**項**

- 透過支援小組開始進行商業理由的工作。
- 每月符合支援小組的需求（或視需要），直到策略小組可以遵循 go/no go 決策，以繼續進行雲端採用。

**支援交付後完成的指引：**

- [商業理由](../../strategy/cloud-migration-business-case.md)：商業理由可做為雲端採用的整體財務計畫的高階觀點。 這可能是初步的預算工作的好來源。

## <a name="step-7-support-adoption-through-a-regular-rhythm"></a>步驟7：透過一般節奏的支援採用

在雲端策略小組同意「向前邁進」策略決策之後，小組可以轉換成較不常且較不頻繁的節奏。 小組的期望也會在此階段轉移。 當旅程從策略性定義移至採用成果（計畫、準備、採納）後，策略小組預期會著重于優先順序和策略性支援。

**項**

- **優先順序：** 當現有的數位資產合理化時，策略小組將協助建立一波遷移或創新的優先順序，協助技術執行團隊專注于驅動最大商業價值的動作。
- **評估風險：** 隨著雲端採用的成長，新的採用形式將會暴露新的風險。 策略小組負責協助評估這些新的風險。 策略小組的期望是要評估新的風險，並判斷企業是否能夠容忍風險，或需要防止風險的原則。
- **審查預算和支出：** 隨著雲端採用的增加，將會針對組合中的各種工作負載進行預算。 雲端策略小組會每月查看實際花費的預算，以找出需要解決的問題。 及早偵測和解決預算變更，可以避免未來採用生命週期中的貼紙。
- **商務規劃：** 當採用小組完成遷移或創新工作時，將需要額外的商務規劃，以最大化新技術解決方案的報酬。 這可能包括終端使用者訓練、商務程式修改，或其他後續採用活動。
- **主管支援：** 雲端採用會導致組織變更。 這在 IT 組織內最是可見的。 有時候，各種小組或小組成員可能需要策略小組的額外支援，以瞭解變更、開發新技能，並瞭解如何在新模型中進行最佳操作。

**支援交付後完成的指引：**

- [增量合理化](../../digital-estate/rationalize.md)：合理化的 agile 方法，適當地配合晚期繫結的技術決策。
- [合理化的五個 Rs](../../digital-estate/5-rs-of-rationalization.md)：瞭解各種合理化選項。

## <a name="whats-next"></a>後續步驟

策略和規劃很重要。 不過，沒有雲端採用功能就不會採取任何動作。 開始採用之前，請先瞭解這項重要功能的角色。

使用採用小組或負責此功能的個人，將您的策略與[雲端採用功能](../../organize/cloud-adoption.md)進行配合。

瞭解如何藉由開發跨小組的矩陣，識別負責、參與、諮詢和通知（RACI）的合作物件，來協調各小組的責任。 下載並修改[RACI 試算表範本](https://archcenter.blob.core.windows.net/cdn/fusion/management/raci-template.xlsx)。
